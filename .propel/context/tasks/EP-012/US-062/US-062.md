# US-062: Export De-identified Encounter Data to BigQuery Nightly

> **Epic:** EP-012 — Analytics & KPI Reporting | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** hospital research analyst,
**I want to** access de-identified encounter data in BigQuery for longitudinal research and population health analysis,
**So that** the hospital can conduct research on care transition outcomes without exposing PHI, supporting both BO-07 (compliance) and future quality improvement initiatives.

---

## Business Value

BigQuery de-identified export (DR-017) creates a compliant analytics data warehouse without HIPAA risk. It separates operational PHI data from analytical data, enabling data science teams to build predictive models and conduct research on large datasets without touching production PHI. This is a regulatory requirement for research use under HIPAA Safe Harbor.

---

## Acceptance Criteria

### Scenario 1: Nightly export runs and BigQuery dataset is updated
**Given** the nightly export Cloud Run job runs at 02:00 UTC
**When** the job completes
**Then** the BigQuery `smarthandoff.encounters_deidentified` dataset contains rows for all encounters completed in the previous day; the export runtime is logged to Cloud Logging.

### Scenario 2: PHI fields are excluded from export
**Given** the de-identification pipeline runs
**When** the BigQuery dataset schema is inspected
**Then** the following columns are absent: `mrn`, `first_name`, `last_name`, `dob`, `phone`, `email`; only de-identified fields (encounter_id_hash, admit_date, discharge_date, diagnosis_codes, risk_score, unit, los_days) are present.

### Scenario 3: Export is idempotent — re-runs do not duplicate rows
**Given** the nightly job ran successfully last night
**When** the job is manually re-run for the same date
**Then** BigQuery rows for that date are REPLACED (not appended), preventing duplicate records.

### Scenario 4: Export failure triggers alerting
**Given** the BigQuery export job fails due to a transient error
**When** the job exits with a non-zero status
**Then** a Cloud Monitoring alert fires; the failure is logged with error details; the data team is notified via email.

---

## Definition of Done

- [ ] Cloud Run nightly export job (Python script): queries Cloud SQL encounter data, applies de-identification, writes to BigQuery
- [ ] De-identification: exclude PHI columns; hash `encounter_id` with SHA-256 to prevent linkage; include only safe fields per HIPAA Safe Harbor
- [ ] BigQuery schema: `encounter_id_hash`, `admit_date`, `discharge_date`, `primary_diagnosis_code`, `risk_score`, `risk_tier`, `unit`, `los_days`, `discharge_disposition`, `readmitted_30d`
- [ ] Idempotent write: `WRITE_TRUNCATE` for the date partition on each run
- [ ] Cloud Scheduler trigger: `0 2 * * *` UTC
- [ ] Cloud Monitoring alert on job failure
- [ ] Code reviewed and Data Privacy Officer reviewed

---

## Technical Notes

- BigQuery client: `google-cloud-bigquery` Python SDK
- Partitioned by `admit_date` — use `TimePartitioning(type_=TimePartitioningType.DAY, field="admit_date")`
- De-identification: `encounter_id_hash = SHA-256(encounter_id + salt)` — salt rotated monthly
- Test: run export on synthetic data in dev; verify PHI fields absent via BigQuery `INFORMATION_SCHEMA.COLUMNS`
