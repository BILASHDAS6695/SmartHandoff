# US-043: Build AI Chatbot with Scoped Discharge Q&A Response

> **Epic:** EP-008 — Patient Communication Agent & Chatbot | **Sprint:** 2 | **Points:** 8 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** patient (Pat) at home after discharge,
**I want to** ask questions about my discharge instructions and receive a response within 3 seconds, with answers limited to information from my own discharge document,
**So that** I get accurate, personalised answers about my care without confusion from generic internet advice.

---

## Business Value

24/7 chatbot availability (FR-060) fills the critical post-discharge support gap when clinical staff are unavailable. Scoping responses to the patient's own documents (FR-061) prevents dangerous generic medical advice. The 3-second response SLA (TR-006) is essential for mobile browser usability — exceeding this causes patients to abandon the chatbot.

---

## Acceptance Criteria

### Scenario 1: Chatbot responds within 3 seconds for 95% of queries
**Given** a patient sends a question via `POST /api/v1/chat`
**When** Gemini Flash processes the request
**Then** a response is returned within 3 seconds for 95% of test queries (p95 latency measured in load test of 100 concurrent users).

### Scenario 2: Response is scoped to patient's own discharge documents only
**Given** patient Pat is asking about medication after Encounter `ENC-001`
**When** the context window is assembled
**Then** the system prompt explicitly restricts the LLM: "You may only answer questions based on the discharge instructions provided. If the answer is not in the instructions, say you don't know and suggest calling the hospital." — the response references only Pat's own documents.

### Scenario 3: Patient cannot access another patient's data via chat
**Given** patient Pat has a JWT scoped to `encounter_id=ENC-001`
**When** the chat API is called with `encounter_id=ENC-002` (a different patient)
**Then** the API returns `403 Forbidden`; no data from ENC-002 is included in any response.

### Scenario 4: Context window respects 8K token limit with FIFO pruning
**Given** a conversation has 15 messages exceeding the 2K conversation history limit
**When** the 16th message is processed
**Then** the oldest messages are pruned from the history to maintain the 2K limit; the system prompt (2K) and discharge context (4K) are preserved without pruning.

---

## Definition of Done

- [ ] `ChatbotAPI` FastAPI service: `POST /api/v1/chat` endpoint accepting `{message, encounter_id, session_id}`
- [ ] JWT validation: `encounter_id` in request must match the `encounter_id` claim in patient JWT (scope enforcement)
- [ ] Context assembly: system prompt (2K) + discharge summary (4K, truncated) + conversation history (2K, FIFO-pruned)
- [ ] Vertex AI Gemini Flash call with 3-second timeout and structured output
- [ ] 3-second timeout: return graceful fallback message on timeout (not an exception)
- [ ] Conversation history stored in Redis: `conversation-history:{encounter_id}:{session_id}` (TTL=24h)
- [ ] Performance test: p95 response latency <3 seconds at 100 concurrent users
- [ ] Unit tests: scope enforcement, context assembly, FIFO pruning, timeout fallback
- [ ] Code reviewed and approved

---

## Technical Notes

- Use Gemini Flash (`gemini-1.5-flash`) for latency; Pro is too slow for 3s SLA
- FIFO pruning: maintain a deque of last 10 messages; when total tokens exceed 2K, drop oldest
- Session ID: UUID generated client-side; `session_id` is NOT the patient ID — allows multiple sessions per encounter
- No PHI should appear in LLM system prompt parameters that are logged; audit log only logs `encounter_id` and message timestamp
