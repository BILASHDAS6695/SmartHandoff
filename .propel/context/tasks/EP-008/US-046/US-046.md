# US-046: Store Encrypted Chatbot Transcripts Against Encounter

> **Epic:** EP-008 — Patient Communication Agent & Chatbot | **Sprint:** 2 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Compliance Officer,
**I want to** ensure all chatbot conversations are stored encrypted against the patient's encounter with urgency flags preserved,
**So that** clinical staff can review the post-discharge conversation history for care quality audits and any urgency signals are permanently recorded.

---

## Business Value

HIPAA requires retention of patient communications. Chatbot transcripts containing patient symptom disclosures are clinical communications that must be retained for 7 years. Urgency flag preservation enables retrospective review of cases where a patient signalled deterioration, supporting quality improvement and malpractice defensibility.

---

## Acceptance Criteria

### Scenario 1: Every chatbot message persisted to transcript table
**Given** a patient sends 5 messages in a chatbot session
**When** the session completes
**Then** 5 `chatbot_transcript` records exist in the DB: each with `encounter_id`, `message` (AES-256-GCM encrypted), `role` (PATIENT or ASSISTANT), `timestamp`, `urgency_flag`, and `escalated` fields.

### Scenario 2: Urgency-flagged messages have urgency_flag=True
**Given** a message triggered urgency detection
**When** the transcript is stored
**Then** that message's `chatbot_transcript.urgency_flag=True` and `escalated=True` (if escalation was published).

### Scenario 3: Transcript message content is encrypted in DB
**Given** a transcript record with `message="chest pain"` was stored
**When** a direct SQL query `SELECT message FROM chatbot_transcript WHERE id=<id>` is executed
**Then** the returned value is ciphertext (AES-256-GCM encrypted), not the plaintext "chest pain".

### Scenario 4: Clinical staff can retrieve transcript for an encounter
**Given** a nurse with a valid staff JWT calls `GET /api/v1/encounters/{id}/chat-transcript`
**When** the API is called
**Then** the decrypted transcript is returned as a chronologically ordered list of messages; patient JWT holders can read only their own encounter transcript.

---

## Definition of Done

- [ ] `chatbot_transcript` ORM table with AES-256-GCM `EncryptedString` TypeDecorator on `message` field
- [ ] Transcript persistence: every message (patient + assistant) stored after each exchange
- [ ] `urgency_flag` and `escalated` boolean fields on `chatbot_transcript`
- [ ] `GET /api/v1/encounters/{id}/chat-transcript` endpoint: staff JWT (any encounter) + patient JWT (own encounter only)
- [ ] Audit log entry created for each transcript access (BR-012)
- [ ] Unit tests: encryption verification (raw DB value is ciphertext), urgency flag persistence, scope enforcement
- [ ] Code reviewed and approved
