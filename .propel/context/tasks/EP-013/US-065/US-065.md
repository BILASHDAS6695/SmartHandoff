# US-065: Generate and Deliver OTP via Twilio Verify

> **Epic:** EP-013 — Notification & Communication Infrastructure | **Sprint:** 2 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** patient accessing the patient portal,
**I want to** receive a 6-digit one-time code by SMS within 30 seconds of requesting it, so I can authenticate to view my discharge instructions,
**So that** I can quickly and securely access my care information without needing to remember a password.

---

## Business Value

OTP delivery is the authentication gateway for all patient portal access (EP-010). Without a reliable OTP service, no patient can access their discharge instructions or chatbot. Twilio Verify is specifically designed for OTP use cases and provides delivery tracking and anti-fraud features not available with raw Twilio SMS.

---

## Acceptance Criteria

### Scenario 1: OTP delivered via Twilio Verify within 30 seconds
**Given** `POST /api/v1/auth/patient/otp` is called with a valid portal token
**When** the OTP request is processed
**Then** Twilio Verify sends an SMS with the 6-digit code within 30 seconds; the OTP hash is stored in Redis with a 10-minute TTL; the API returns `202 Accepted`.

### Scenario 2: OTP rate limit: 5 per phone per hour
**Given** a phone number `+1234567890` has requested 5 OTPs in the last 60 minutes
**When** a 6th OTP request is made
**Then** the API returns `429 Too Many Requests` with `Retry-After: 3600`; no OTP is generated or sent to Twilio.

### Scenario 3: Incorrect OTP returns 401 with remaining attempts counter
**Given** a patient enters an incorrect OTP
**When** `POST /api/v1/auth/patient/verify` is called
**Then** the response is `401 Unauthorized` with body `{"error": "invalid_otp", "attempts_remaining": 2}`; a Redis counter increments; after 3 failed attempts, the OTP is invalidated.

### Scenario 4: OTP expired after 10 minutes returns clear error
**Given** an OTP was generated 11 minutes ago
**When** the patient submits it via `POST /api/v1/auth/patient/verify`
**Then** the response is `401 Unauthorized` with `{"error": "otp_expired", "message": "Please request a new code"}`; no JWT is issued.

---

## Definition of Done

- [ ] `POST /api/v1/auth/patient/otp`: Twilio Verify `create_verification()` call; OTP hash stored in Redis (TTL=600s); rate limit check
- [ ] OTP hash: `bcrypt(otp, rounds=10)` stored in Redis — NOT plaintext
- [ ] `POST /api/v1/auth/patient/verify`: fetch hash from Redis, `bcrypt.checkpw()`, issue JWT on success
- [ ] Rate limit: Redis counter `otp_rate:{phone_hash}` (hash phone number, don't store plaintext) with TTL=3600s
- [ ] Failed attempts: Redis counter `otp_failures:{otp_key}` (max 3); OTP invalidated after 3 failures
- [ ] Twilio Verify service SID from Secret Manager
- [ ] Unit tests: rate limit, expiry, failed attempts, successful verification
- [ ] Code reviewed and approved

---

## Technical Notes

- Twilio Verify API: `client.verify.v2.services(VERIFY_SID).verifications.create(to=phone, channel="sms")`
- Phone number hashing: `SHA-256(phone_number + salt)` for Redis key — prevents phone enumeration from Redis dump
- Redis OTP key: `otp:{SHA-256(portal_token)}` — links OTP to the portal token, not directly to phone
- bcrypt rounds: 10 (standard); OTP verification is low-traffic, so bcrypt cost is acceptable
