# US-067: Provide Notification Audit Log API with Patient Opt-Out Support

> **Epic:** EP-013 — Notification & Communication Infrastructure | **Sprint:** 2 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** care team member,
**I want to** see the notification delivery history for a patient's encounter — including delivery status, timestamp, and channel — and as a patient, I want to be able to opt out of non-urgent notifications,
**So that** care teams can verify patients received critical communications and patients retain control over their communication preferences.

---

## Business Value

Notification audit logs provide clinical evidence that a patient was informed (e.g. appointment reminder was delivered 3 days before the follow-up). This is relevant for readmission disputes and quality review. Patient opt-out (AIR-043) respects patient autonomy and reduces SMS/email fatigue that would otherwise lead to patients ignoring genuinely important messages.

---

## Acceptance Criteria

### Scenario 1: Care team sees notification history for encounter
**Given** a nurse calls `GET /api/v1/notifications?encounter_id={id}`
**When** the API is called with a valid staff JWT
**Then** the response lists all notifications for that encounter: `type`, `channel`, `sent_at`, `delivery_status` (SENT/DELIVERED/FAILED/OPTED_OUT), `template_name`; no PHI in notification content fields.

### Scenario 2: Opt-out flag prevents non-urgent notifications
**Given** patient `patient-123` has `notification_opt_out=True`
**When** a `medication_reminder` (non-urgent) notification is triggered
**Then** the notification service skips sending; the `notification` record is created with `delivery_status=OPTED_OUT`; no SMS or email is dispatched.

### Scenario 3: Urgent notifications bypass opt-out
**Given** patient `patient-123` has `notification_opt_out=True`
**When** a `CARE_TEAM_URGENCY_ALERT` notification (urgency_override=True) is triggered
**Then** the notification is sent despite the opt-out flag; the `notification` record is created with `delivery_status=SENT`, `urgency_override=True`; no opt-out suppression occurs.

### Scenario 4: Patient can set opt-out preference via portal API
**Given** a patient with a valid portal JWT calls `PATCH /api/v1/portal/preferences`
**When** the request body is `{"notification_opt_out": true}`
**Then** `patient.notification_opt_out=True` is persisted; the API returns `200 OK`; future non-urgent notifications are suppressed.

---

## Definition of Done

- [ ] `GET /api/v1/notifications?encounter_id={id}` endpoint: staff JWT required; returns delivery log
- [ ] `PATCH /api/v1/portal/preferences` endpoint: patient JWT required; updates `notification_opt_out`
- [ ] Opt-out check in notification service: `if patient.notification_opt_out and not msg.urgency_override: skip()`
- [ ] `urgency_override` boolean field on notification Pub/Sub message schema
- [ ] `notification` table `delivery_status` enum: `PENDING`, `SENT`, `DELIVERED`, `FAILED`, `OPTED_OUT`
- [ ] Audit log entry created for every notification delivery attempt (BR-012 compliance)
- [ ] Unit tests: opt-out suppression, urgency bypass, patient preference update, staff log query
- [ ] Code reviewed and approved

---

## Technical Notes

- Notification log API: use read replica (US-009) for query performance
- PHI minimisation in notification log: `recipient_phone_hash` (not plaintext phone), `recipient_email_hash`; only `encounter_id` and patient's own data accessible
- Opt-out persisted on `patient` table (US-006 ORM model) — not notification-table
- Urgency override flag must be set by the sending agent (not overrideable by patient)
