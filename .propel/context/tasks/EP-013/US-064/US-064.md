# US-064: Build Notification Service for SMS and Email Dispatch

> **Epic:** EP-013 — Notification & Communication Infrastructure | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Backend Engineer,
**I want to** build a Notification Service that reads notification requests from Pub/Sub `notification-requests`, dispatches SMS via Twilio and email via SendGrid using Dynamic Templates, tracks delivery status, and enforces idempotency,
**So that** patients and staff reliably receive care-critical notifications without duplicate sends.

---

## Business Value

The Notification Service is the communication backbone that connects AI agent decisions to real human action. Without it, discharge check-ins (EP-007), pharmacist alerts (EP-005), ED boarding alerts (EP-006), and OTP authentication (EP-010) cannot reach their recipients. Idempotency prevents duplicate SMS during Pub/Sub message redelivery — critically important for high-risk clinical alerts.

---

## Acceptance Criteria

### Scenario 1: SMS sent within 30 seconds of Pub/Sub notification-requests message
**Given** a `notification-requests` Pub/Sub message with `type=SMS`, `phone=+1234567890`, `template=medication_reminder`
**When** the notification service processes the message
**Then** an SMS is sent via Twilio within 30 seconds; a `notification` DB record with `status=SENT` and `twilio_message_sid` is created.

### Scenario 2: Idempotency key prevents duplicate SMS
**Given** a notification with `idempotency_key=NOTIF-001` was already sent
**When** the same Pub/Sub message is redelivered (due to Pub/Sub at-least-once delivery)
**Then** the service detects `idempotency_key=NOTIF-001` already exists in DB; no SMS is sent; the Pub/Sub message is acknowledged.

### Scenario 3: Twilio delivery webhook updates status to DELIVERED
**Given** Twilio sends a delivery webhook `POST /webhooks/twilio/status` with `MessageSid` and `MessageStatus=delivered`
**When** the webhook is processed
**Then** the `notification.status` updates to `DELIVERED`; the webhook signature is validated (prevent spoofing).

### Scenario 4: Failed delivery retried 3 times with backoff
**Given** Twilio returns a transient error (HTTP 503) on first send attempt
**When** the notification service retries
**Then** retry 2 is attempted after 30 seconds, retry 3 after 60 seconds; if all 3 fail, `notification.status=FAILED` and a `CARE_TEAM_ALERT` is published for manual follow-up.

## Definition of Done

- [ ] `NotificationService` Cloud Run service: Pub/Sub pull subscription on `notification-requests`
- [ ] Twilio SMS: `twilio.rest.Client.messages.create(to, from, template_id, substitutions)`
- [ ] SendGrid email: `SendGridAPIClient.send(message)` with Dynamic Template rendering
- [ ] `notification` ORM table: `idempotency_key` (unique), `type`, `status`, `twilio_message_sid`, `sendgrid_message_id`, `phone_or_email`
- [ ] Idempotency check: `INSERT ... ON CONFLICT (idempotency_key) DO NOTHING`
- [ ] Twilio webhook: `POST /webhooks/twilio/status` with Twilio signature validation (`X-Twilio-Signature` header)
- [ ] Retry: APScheduler retries failed notifications 3× with 30s/60s/120s backoff
- [ ] Opt-out flag: `patient.notification_opt_out=True` → skip send + log
- [ ] Unit tests: idempotency, retry logic, Twilio webhook validation
- [ ] Code reviewed and approved

---

## Technical Notes

- Twilio credentials: `account_sid` and `auth_token` from Secret Manager
- SendGrid credentials: `api_key` from Secret Manager
- Twilio webhook signature validation: `twilio.request_validator.RequestValidator.validate()` with auth token
- Pub/Sub message schema: `{idempotency_key, type, priority, recipient_id, template, substitutions}`
