# US-004: JWT Token Blocklist for Instant User Deprovisioning

> **Epic:** EP-011 — Security, Auth & HIPAA Compliance | **Sprint:** 2 | **Points:** 3 | **Priority:** Must Have

## User Story

**As an** IT admin,
**I want** to deprovision a user and have their access revoked immediately — even if their JWT hasn't expired yet,
**so that** a departing staff member's access is terminated the moment HR notifies IT, not hours later when their token expires.

## Acceptance Criteria

- [ ] **AC-1:** User deprovisioning `DELETE /api/v1/admin/users/{id}` → sets `AppUser.active=false` → adds all current JTIs (JWT IDs) for that user to Redis blocklist with TTL = remaining JWT expiry time
- [ ] **AC-2:** `RBACEnforcer` middleware checks `GET redis:blocklist:{jti}` on every authenticated request; blocklisted JTI → 401 Unauthorized with `error=token_revoked`; Redis lookup adds ≤5ms overhead
- [ ] **AC-3:** SCIM 2.0 provisioning: `POST /api/v1/admin/users/provision` creates `AppUser` and syncs to Identity Provider; new user receives onboarding SMS with SSO link and MFA setup instructions
- [ ] **AC-4:** Session timeout: Angular `IdleTimeoutService` fires after 30 minutes of inactivity → calls `POST /api/v1/auth/logout` → adds JTI to blocklist → clears in-memory JWT → redirects to SSO logout
- [ ] **AC-5:** Integration test: login as user → obtain JWT → deprovision user → make authenticated API call with same JWT → verify 401 returned within 1 second of deprovisioning

## Definition of Done

- [ ] Deprovisioning test passes: JWT rejected within 1 second of deprovisioning
- [ ] Redis blocklist TTL verified: expired JTIs auto-deleted from Redis after token expiry
- [ ] SCIM endpoint tested with a mock IdP SCIM POST
