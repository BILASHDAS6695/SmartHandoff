# US-005: Security Headers, Rate Limiting, and Input Validation

> **Epic:** EP-011 — Security, Auth & HIPAA Compliance | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have

## User Story

**As a** security engineer,
**I want** all API responses to include OWASP-recommended security headers, rate limiting, and server-side input validation,
**so that** the API is protected against XSS, clickjacking, injection attacks, and brute-force attempts.

## Acceptance Criteria

- [ ] **AC-1:** Security headers on all API responses: `Strict-Transport-Security: max-age=31536000; includeSubDomains`, `Content-Security-Policy: default-src 'self'`, `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin` — verified via curl on staging API
- [ ] **AC-2:** Rate limiting via `slowapi`: 1,000 requests/minute per authenticated user; 100 requests/minute per IP for public endpoints (auth, OTP) — 429 Too Many Requests with `Retry-After` header returned on breach; rate limit headers included in every response
- [ ] **AC-3:** All API input validated via Pydantic v2 models at FastAPI route boundary; malformed input returns 422 Unprocessable Entity with field-level error details (no stack traces, no PHI in errors)
- [ ] **AC-4:** Parameterised queries enforced via SQLAlchemy ORM — no raw SQL string interpolation; bandit SAST scan confirms zero B608 (SQL injection) findings in CI
- [ ] **AC-5:** Cloud Armor WAF active on load balancer with OWASP Core Rule Set; rules tested in detection mode in staging before enforcement in production

## Definition of Done

- [ ] Security headers verified via `curl -I https://api.staging.smarthandoff.health/api/v1/health`
- [ ] Rate limit test: 1,001 requests in 60 seconds from one IP → 429 returned
- [ ] bandit + SAST scan in CI passes with zero HIGH severity findings

## Dependencies

- EP-TECH US-001 (Cloud Armor WAF provisioned)
- EP-TECH US-002 (CI/CD pipeline for bandit scan)
