# US-056: Implement OIDC/MFA Staff Authentication with JWT Issuance

> **Epic:** EP-011 — Security, Auth & HIPAA Compliance | **Sprint:** 1 | **Points:** 8 | **Priority:** Must Have
> **Status:** Done | **Date:** 2026-07-22

---

## User Story

**As a** clinical staff member,
**I want to** log in via my hospital's identity provider using OIDC with MFA enforced, and receive an application JWT,
**So that** my identity is strongly verified before I access any patient data and the system is compliant with HIPAA access control requirements.

---

## Business Value

MFA enforcement (SEC-001, AIR-033) is the primary defence against credential theft in healthcare environments. Credential phishing attacks on hospital staff are the #1 vector for healthcare data breaches. OIDC federation with the hospital's existing IdP eliminates password management overhead and ensures seamless deprovisioning when staff leave.

---

## Acceptance Criteria

### Scenario 1: Staff login redirects to OIDC and issues JWT on MFA completion
**Given** a staff member navigates to the SmartHandoff login page
**When** they complete the OIDC redirect, enter MFA code, and are redirected back
**Then** SmartHandoff validates the OIDC `id_token`, checks `amr` claim contains `mfa`, and issues an application JWT with role and unit claims; the JWT is stored in Angular memory only (not localStorage).

### Scenario 2: Login rejected if MFA not present in amr claim
**Given** a staff member's OIDC token has `amr: ["password"]` (no MFA)
**When** the JWT validation middleware processes the token
**Then** the response is `401 Unauthorized` with message "MFA required"; no application JWT is issued.

### Scenario 3: JWKS cache refreshes every hour
**Given** the OIDC JWKS endpoint returns a JSON Web Key Set
**When** the SmartHandoff backend validates an ID token
**Then** the JWKS is fetched from cache if it was last refreshed <1 hour ago; otherwise a fresh fetch from the IdP is made; the JWKS is not fetched on every request.

### Scenario 4: JWT stored in Angular memory only
**Given** a staff member successfully logs in
**When** `localStorage`, `sessionStorage`, and `document.cookie` are inspected in the browser
**Then** no JWT token is found in any of these storage locations; the token is only in Angular's `AuthService` memory.

---

## Definition of Done

- [x] FastAPI OIDC middleware: JWKS cache (1h TTL), `id_token` signature validation, `amr` claim check
- [x] Application JWT issuance: `sub=user_id`, `role`, `units`, `exp=8h`, signed with HS256 (Secret Manager key)
- [x] JWT claims mapping from OIDC claims: `sub → user_id`, `groups → role`, custom claim `units`
- [x] Angular `AuthService`: stores JWT in private class field; exposes `getToken()` method
- [x] `AuthGuard`: `canActivate()` checks JWT expiry from `AuthService`; redirects to login if expired
- [x] Angular login callback component: handles OIDC redirect, extracts code, calls token exchange API
- [x] Unit tests: `amr` validation, JWKS cache, JWT claim mapping
- [ ] Code reviewed and Security Engineer reviewed

---

## Technical Notes

- OIDC discovery: `GET {idp_base_url}/.well-known/openid-configuration` for JWKS URI
- `python-jose` library for JWT validation in FastAPI middleware
- JWKS cache: module-level `TTLCache(maxsize=1, ttl=3600)` from `cachetools`
- Angular: `HubConnectionBuilder.withUrl(url, {accessTokenFactory: () => authService.getToken()})`
