# US-060: Implement SCIM 2.0 User Provisioning and Deprovisioning

> **Epic:** EP-011 — Security, Auth & HIPAA Compliance | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Done | **Date:** 2026-07-22

---

## User Story

**As an** IT administrator,
**I want to** have user accounts automatically created and deprovisioned in SmartHandoff via SCIM 2.0 whenever staff are onboarded or offboarded in the hospital's identity provider,
**So that** user lifecycle management is automated and immediate access revocation prevents data breaches when staff leave.

---

## Business Value

Manual user provisioning creates a security gap: new staff can't access the system promptly (workflow delay) and departed staff remain in the system too long (security gap). SCIM 2.0 automation eliminates both delays, directly supporting BR-013 (session management) and HIPAA workforce management requirements (BO-07 compliance score 100%).

---

## Acceptance Criteria

### Scenario 1: SCIM POST creates new app_user record
**Given** the hospital IdP sends a SCIM 2.0 `POST /api/v1/admin/scim/Users` request for a new nurse
**When** the SCIM endpoint processes the request
**Then** an `app_user` record is created with: `email`, `display_name`, `role=NURSE`, `unit` (from SCIM `enterpriseUser.department`), `scim_id` from the IdP.

### Scenario 2: SCIM DELETE triggers immediate JWT blocklist
**Given** a SCIM `DELETE /api/v1/admin/scim/Users/{id}` request is received for a deprovisioned staff member
**When** the endpoint processes the request
**Then** `app_user.deprovisioned_at` is set; any active JWTs for that user are added to the Redis blocklist within 1 second; subsequent API calls with their JWT return `401 Unauthorized`.

### Scenario 3: SCIM requests are authenticated
**Given** an unauthenticated SCIM `POST /api/v1/admin/scim/Users` request is made
**When** the request reaches the SCIM endpoint
**Then** the response is `401 Unauthorized`; the SCIM bearer token in the `Authorization` header is required and validated against the stored SCIM client secret.

### Scenario 4: SCIM PATCH updates user role
**Given** a SCIM `PATCH /api/v1/admin/scim/Users/{id}` request changes a user's department from `Nursing` to `Pharmacy`
**When** the endpoint processes the request
**Then** `app_user.role` updates from `NURSE` to `PHARMACIST` via the department-to-role mapping; the change is logged to `audit_log`.

## Definition of Done

- [x] SCIM 2.0 endpoints: `POST`, `GET`, `PATCH`, `PUT`, `DELETE` `/api/v1/admin/scim/Users`
- [x] SCIM bearer token authentication: configurable SCIM client secret in Secret Manager
- [x] Department → role mapping: `config/scim_role_mapping.yaml`
- [x] SCIM DELETE: calls same `deprovisioning_service.deprovision_user(user_id)` as manual deprovision (US-059)
- [x] SCIM compliance: return correct SCIM 2.0 response schemas (RFC 7643)
- [x] Unit tests: create, update, delete, auth failure
- [x] Code reviewed and Security Engineer reviewed

---

## Technical Notes

- SCIM 2.0 user schema: `id`, `userName`, `name.familyName`, `name.givenName`, `emails[0].value`, `active`, `urn:ietf:params:scim:schemas:extension:enterprise:2.0:User.department`
- Department-to-role mapping: `{Nursing: NURSE, Pharmacy: PHARMACIST, Medicine: PHYSICIAN, BedManagement: BED_MANAGER, ...}`
- `scim_id` stored on `app_user` for IdP-to-SmartHandoff cross-reference
- SCIM bearer token: separate from staff JWTs; long-lived (90-day rotation); stored in Secret Manager
