# US-059: Implement JWT Token Blocklist and 30-Minute Session Timeout

> **Epic:** EP-011 — Security, Auth & HIPAA Compliance | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have
> **Status:** Done | **Date:** 2026-07-22

---

## User Story

**As a** Security Engineer,
**I want to** ensure deprovisioned user JWTs are immediately rejected via a Redis blocklist, and all sessions automatically expire after 30 minutes of inactivity,
**So that** terminated employees lose access instantly and unattended workstations with an open session cannot be used to access PHI.

---

## Business Value

Without immediate token revocation, a terminated employee retains access for the remaining JWT lifetime (up to 8 hours). This is a critical HIPAA security gap. The 30-minute idle timeout (BR-013) is a standard healthcare security control that prevents tailgating attacks on unattended clinical workstations.

---

## Acceptance Criteria

### Scenario 1: Deprovisioned user JWT rejected within 1 second
**Given** user `user-123` has an active JWT and is deprovisioned
**When** `DELETE /api/v1/admin/users/user-123` is called
**Then** the JWT is added to the Redis blocklist immediately; the next API call with that JWT returns `401 Unauthorized` within 1 second.

### Scenario 2: Active users are not affected by blocklist check
**Given** 1,000 concurrent staff users are making API calls
**When** the JWT middleware checks the Redis blocklist on each request
**Then** no latency impact >5ms is observed on API response time (p99) for non-blocklisted tokens; Redis TTL-based cleanup prevents unbounded blocklist growth.

### Scenario 3: 30-minute idle timeout triggers logout
**Given** a nurse's Angular app has had no API activity for 30 minutes
**When** the `IdleTimeoutService` fires
**Then** the Angular app clears the in-memory JWT, displays a "Session expired" modal, and redirects to the login page; the next API call (if attempted) returns `401 Unauthorized`.

### Scenario 4: User logout adds JWT to blocklist
**Given** a staff member clicks "Log Out"
**When** `POST /api/v1/auth/logout` is called with the current JWT
**Then** the JWT `jti` (JWT ID) claim is added to the Redis blocklist; the Angular `AuthService` clears the in-memory JWT; subsequent requests with the old JWT return `401 Unauthorized`.

---

## Definition of Done

- [x] JWT blocklist: Redis SADD `jwt_blocklist:{jti}` on logout/deprovision; TTL = remaining JWT lifetime
- [x] FastAPI JWT middleware: `if redis.sismember("jwt_blocklist", jti): raise HTTPException(401)`
- [x] `DELETE /api/v1/admin/users/{id}` deprovisioning endpoint: adds JWT to blocklist + sets `app_user.deprovisioned_at`
- [x] `POST /api/v1/auth/logout` endpoint: blocklist current JWT + 200 OK
- [x] Angular `IdleTimeoutService`: Rxjs timer resets on any mousemove/keypress/scroll; fires logout at 30 minutes
- [x] "Session expired" modal: `MatDialog` with auto-dismiss and redirect to login
- [x] Unit tests: blocklist lookup, deprovisioning flow, idle timeout event
- [x] Code reviewed and Security Engineer reviewed

---

## Technical Notes

- JWT `jti` claim: UUID generated at JWT issuance time; stored as `jti` in JWT payload
- Redis blocklist key TTL: `jwt_exp - current_time` seconds (auto-expires when token would have expired anyway)
- Idle timer: Angular `fromEvent(document, 'mousemove')` merged with keypress/scroll events; `switchMap(() => timer(1800000))` fires after 30 minutes of no events
- Security: blocklist check must happen AFTER signature validation — invalid signatures still rejected first
