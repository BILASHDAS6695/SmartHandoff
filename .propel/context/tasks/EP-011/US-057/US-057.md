# US-057: Enforce RBAC for All 7 Clinical Roles via FastAPI Dependency

> **Epic:** EP-011 — Security, Auth & HIPAA Compliance | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have
> **Status:** Done | **Date:** 2026-07-14

---

## User Story

**As a** Security Engineer,
**I want to** implement role-based access control as a FastAPI dependency that enforces resource-action permissions for all 7 clinical roles,
**So that** a nurse cannot access pharmacist functions, a patient cannot access staff data, and HIPAA minimum-necessary access is enforced at the API layer.

---

## Business Value

RBAC is the primary access control mechanism for HIPAA §164.312(a)(1) (Access Control). Without it, any authenticated user could call any API endpoint — a nurse could approve discharge summaries, or a pharmacist could modify bed assignments. Role-based separation maintains clinical workflow integrity and audit defensibility.

---

## Acceptance Criteria

### Scenario 1: Nurse cannot access pharmacist alert resolution endpoint
**Given** a nurse JWT with `role=NURSE`
**When** `PATCH /api/v1/alerts/{id}/resolve` is called
**Then** the response is `403 Forbidden`; the alert is unchanged; an audit log entry records the unauthorised access attempt.

### Scenario 2: Correct role grants access
**Given** a pharmacist JWT with `role=PHARMACIST`
**When** `PATCH /api/v1/alerts/{id}/resolve` is called
**Then** the request succeeds (2xx); the alert is updated; an audit log entry records the successful action.

### Scenario 3: Permission matrix defines 7 roles × all resources
**Given** the RBAC permission matrix is loaded from `config/rbac_permissions.yaml`
**When** the FastAPI application starts
**Then** all 7 roles (`PHYSICIAN`, `NURSE`, `PHARMACIST`, `BED_MANAGER`, `CARE_MANAGER`, `ADMIN`, `PATIENT`) are defined with their resource-action permissions; missing role → startup validation error.

### Scenario 4: Patient role cannot access staff-facing endpoints
**Given** a patient JWT with `role=PATIENT`
**When** `GET /api/v1/patients` (staff patient list) is called
**Then** the response is `403 Forbidden`; patient role only has access to own-encounter portal endpoints.

## Definition of Done

- [x] `RBACMiddleware` as a FastAPI `Depends()` function: `require_permission(resource, action)` → checks `jwt.role` against RBAC matrix
- [x] RBAC permission matrix in `config/rbac_permissions.yaml`: 7 roles × resources × actions (READ/WRITE/APPROVE/RESOLVE)
- [x] All protected API endpoints decorated with `Depends(require_permission("resource", "action"))`
- [x] `403 Forbidden` response on permission denial; unauthorised access attempts logged to `audit_log`
- [x] Startup validation: RBAC config file validated on application start (missing role → refuse startup)
- [x] Unit tests: each role boundary (6 roles × 3+ endpoints each)
- [x] Code reviewed and Security Engineer reviewed

---

## Technical Notes

- `require_permission` FastAPI dependency: `async def require_permission(resource: str, action: str) -> Callable`
- RBAC matrix YAML: `{PHYSICIAN: {document: [read, write, approve], alert: [read], ...}}`
- Permission check: `if action not in rbac_matrix[jwt_role][resource]: raise HTTPException(403)`
- PATIENT role: hardcoded to own-encounter-only access (no YAML config — immutable security boundary)
