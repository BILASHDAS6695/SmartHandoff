# US-003: RBAC Permission Enforcement Across All API Endpoints

> **Epic:** EP-011 — Security, Auth & HIPAA Compliance | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have

## User Story

**As a** security engineer,
**I want** every API endpoint protected by role-based access control that rejects unauthorised requests with 403,
**so that** a nurse can never access pharmacist data, a patient can never access another patient's records, and data is always minimum-necessary.

## Acceptance Criteria

- [ ] **AC-1:** FastAPI `RBACEnforcer` dependency injection checks JWT `role` claim against a permission matrix per endpoint; rejects with 403 Forbidden if role not permitted
- [ ] **AC-2:** Permission matrix covers all 7 roles (ADMIN, PHYSICIAN, NURSE, PHARMACIST, BED_MANAGER, PATIENT, READ_ONLY) × all resource/action combinations matching the RBAC matrix in design.md §8.3
- [ ] **AC-3:** Data-level filtering: NURSE queries auto-filtered to `WHERE unit={jwt.unit}`; PATIENT queries filtered to `WHERE encounter_id={jwt.encounter_id}`; no application code can bypass this filter without failing the RBACEnforcer
- [ ] **AC-4:** PHYSICIAN approve endpoint (`PATCH /api/v1/documents/{id}/approve`) returns 403 for NURSE, PHARMACIST, BED_MANAGER, and PATIENT roles — verified in integration tests
- [ ] **AC-5:** PHARMACIST medication resolution endpoint (`PATCH /api/v1/alerts/{id}/resolve`) returns 403 for all non-PHARMACIST roles
- [ ] **AC-6:** RBAC permission matrix stored in a config file (not hardcoded); adding a new role requires only config change + tests — no middleware code change

## Definition of Done

- [ ] RBAC matrix integration tests: 35 role × endpoint combinations tested; all return correct 200/403 responses
- [ ] Data-level filter: nurse account API response contains only own-unit encounter data (verified against DB)

## Dependencies

- EP-011 US-001 (JWT with role claims)
- EP-DATA US-001 (DB tables with unit/encounter_id for data-level filtering)
