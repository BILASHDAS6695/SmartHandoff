# US-058: Log All PHI Access to Immutable Audit Trail

> **Epic:** EP-011 — Security, Auth & HIPAA Compliance | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** HIPAA Compliance Officer,
**I want to** ensure every access to patient PHI (reads and writes) is automatically logged to an immutable audit trail with user identity, action type, and IP address,
**So that** any HIPAA audit or breach investigation can produce a complete, tamper-proof access history.

---

## Business Value

HIPAA §164.312(b) mandates hardware, software, and procedural mechanisms to record and examine PHI access. The immutable audit trail (SEC-006) provides the evidentiary foundation for breach investigations, regulatory audits, and internal compliance reviews. Combined with the PostgreSQL RLS from US-008, this creates a court-admissible access record.

---

## Acceptance Criteria

### Scenario 1: PHI endpoint access creates audit log entry
**Given** a nurse calls `GET /api/v1/patients/{id}`
**When** the request is processed
**Then** an `audit_log` entry is created with: `user_id`, `action=READ`, `entity_type=PATIENT`, `entity_id={id}`, `ip_address` (from request headers), `timestamp=UTC`.

### Scenario 2: PHI write actions are logged
**Given** a physician calls `PATCH /api/v1/documents/{id}/approve`
**When** the approval is processed
**Then** an `audit_log` entry is created with `action=APPROVE`, `entity_type=DOCUMENT`, `entity_id={id}`, `user_id=physician_id`.

### Scenario 3: PHI fields not present in Cloud Logging
**Given** the PHI log sanitiser middleware is active
**When** any API handler logs a structured message containing `first_name`, `last_name`, or `mrn`
**Then** the Cloud Logging export shows those field values as `[REDACTED]`; the original values remain in the secure `audit_log` DB table only.

### Scenario 4: Audit log query API returns paginated results
**Given** a compliance admin calls `GET /api/v1/admin/audit?user_id={id}&from={date}&to={date}`
**When** the query executes
**Then** paginated audit log entries are returned; PHI fields in the response are masked based on the caller's role (non-compliance roles see masked names); standard roles see only `entity_type` and `entity_id`.

---

## Definition of Done

- [ ] `AuditLogMiddleware` FastAPI middleware: logs every request to `/api/v1/` with PHI entity paths
- [ ] `audit_log` ORM model: `user_id`, `action` (READ/WRITE/DELETE/APPROVE/REJECT), `entity_type`, `entity_id`, `ip_address`, `user_agent`, `timestamp`
- [ ] PHI sanitiser middleware: strips `first_name`, `last_name`, `mrn`, `dob`, `phone`, `email` from log messages before Cloud Logging emission
- [ ] `GET /api/v1/admin/audit` paginated query endpoint (compliance admin role only)
- [ ] IP address extraction: `request.client.host` with `X-Forwarded-For` header support (Cloud Run proxy-aware)
- [ ] Unit tests: audit entry creation, PHI sanitisation (no PHI in logs), admin query RBAC
- [ ] Code reviewed and Security Engineer reviewed

---

## Technical Notes

- Audit log write uses dedicated `audit_writer` DB role (US-008) — ensures RLS policy is respected
- FastAPI middleware runs after routing — can access `request.state.current_user` for `user_id`
- Log sanitiser: use Python `logging.Filter` subclass that regex-replaces PHI patterns
- For compliance admin role: return unmasked entity IDs but still mask name/MRN in API response (show only to HIPAA Privacy Officer role)
