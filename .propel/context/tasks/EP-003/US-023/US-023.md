# US-023: Generate Context-Aware Handoff Checklist via LLM

> **Epic:** EP-003 — AI Agent Orchestration Framework | **Sprint:** 2 | **Points:** 5 | **Priority:** Should Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** charge nurse receiving a patient handoff,
**I want to** receive an AI-generated handoff checklist tailored to the specific patient's diagnosis, care unit, and transition type,
**So that** I never miss a critical care item specific to this patient's clinical situation, reducing handoff errors.

---

## Business Value

Generic checklist templates miss patient-specific risks. A patient transferring from ICU with ventilator dependency has entirely different handoff items than a post-surgical patient. AI-generated checklists directly reduce the cognitive burden on nurses and eliminate patient-specific omissions that cause adverse events.

---

## Acceptance Criteria

### Scenario 1: Checklist contains ≥3 patient-specific items for discharge scenario
**Given** a discharge (A03) event for a patient with diagnoses: `Type 2 Diabetes`, `Heart Failure`, and medications: `Metformin`, `Furosemide`
**When** the coordinator generates the handoff checklist
**Then** the checklist contains ≥3 items specifically referencing the patient's conditions (e.g. "Verify blood glucose monitoring plan", "Confirm diuretic dose adjustment per discharge orders").

### Scenario 2: Checklist context includes minimum-necessary PHI only
**Given** the LLM prompt is assembled for checklist generation
**When** the prompt template is rendered
**Then** it includes diagnosis codes, unit name, and transition type — NOT the patient's full name, DOB, phone number, or MRN (minimum-necessary PHI enforcement per AIR-021).

### Scenario 3: Checklist stored and accessible via task API
**Given** a handoff checklist has been generated for encounter `ENC-001`
**When** `GET /api/v1/encounters/ENC-001/tasks` is called
**Then** the coordinator `AgentTask` response includes a `checklist` field with the generated items as a structured JSON array.

### Scenario 4: LLM timeout falls back to template checklist
**Given** the Gemini LLM call takes more than 15 seconds
**When** the timeout fires
**Then** a default template checklist (transition-type-specific, not patient-specific) is used; the checklist is flagged as `generated_type=TEMPLATE`; the task still completes successfully.

---

## Definition of Done

- [ ] Jinja2 prompt template for checklist generation: `prompts/checklist.jinja2` with placeholders for diagnosis codes, unit, transition type
- [ ] Vertex AI Gemini call with structured output schema: `{"checklist": [{"item": str, "category": str, "priority": str}]}`
- [ ] Pydantic `HandoffChecklist` model with validation for required fields
- [ ] 15-second timeout on Gemini call; template fallback with pre-defined items per transition type
- [ ] `checklist` field stored as JSONB in `agent_task.metadata` column
- [ ] PHI audit: unit test confirms patient PII fields (`first_name`, `last_name`, `mrn`, `dob`) are absent from prompt before Gemini call
- [ ] Code reviewed and approved

---

## Technical Notes

- Prompt minimum-necessary PHI: use ICD-10 diagnosis codes instead of diagnosis text where possible
- Gemini structured output: use `response_schema` parameter with the `HandoffChecklist` Pydantic JSON schema
- Template fallback items stored in `config/checklist_templates.yaml` keyed by transition type
- Checklist items should include actionable verb: "Verify...", "Confirm...", "Schedule...", "Review..."
