# US-024: Implement LangChain Base Agent Framework with DLQ Handling

> **Epic:** EP-003 — AI Agent Orchestration Framework | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As an** AI/ML Engineer,
**I want to** have a reusable LangChain base agent class that provides Pub/Sub message consumption, structured output with Pydantic schemas, retry with exponential backoff, DLQ handling, and cancellation flag checking,
**So that** all 5 specialist agents (Documentation, Medication Reconciliation, Bed Management, Follow-up Care, Patient Communication) inherit consistent behaviour without duplicating boilerplate.

---

## Business Value

A shared base class enforces consistent error handling, retry behaviour, and cancellation awareness across all agents. Without it, each of the 5 agents would implement these differently, creating 5 independent failure modes. The DLQ handling (TR-015) ensures no clinical message is silently lost.

---

## Acceptance Criteria

### Scenario 1: Base agent consumes and acks Pub/Sub message after success
**Given** a specialist agent extends `BaseAgent` and its `process()` method completes successfully
**When** the Pub/Sub message is processed
**Then** the message is acknowledged; the `AgentTask` `status` is updated to `COMPLETED`; no exception propagates.

### Scenario 2: Base agent nacks message on retryable failure
**Given** the agent's `process()` method raises a `RetryableError` (e.g. DB connection timeout)
**When** the error is caught by the base agent
**Then** the Pub/Sub message is nacked (returned to the subscription); the `AgentTask` retry counter is incremented; backoff delay is applied before the next attempt.

### Scenario 3: Cancellation flag prevents agent from persisting output
**Given** an agent is mid-execution and a cancellation flag is set for the encounter
**When** the agent checks the flag before persisting its result
**Then** the agent exits without writing to the DB; the `AgentTask` is marked `CANCELLED`; the Pub/Sub message is acknowledged (to prevent DLQ).

### Scenario 4: Non-retryable failure sends message to DLQ
**Given** a message fails 5 consecutive delivery attempts
**When** the last attempt fails with a non-retryable error
**Then** the message is moved to the DLQ subscription; a Cloud Monitoring alert fires on `dlq_message_count > 0`; the `AgentTask` is marked `FAILED` with error details.

---

## Definition of Done

- [ ] `BaseAgent` abstract class with: `subscribe()`, `process()` (abstract), `ack()`, `nack()`, `check_cancellation()`, `update_task_status()`
- [ ] Retry decorator: 3 attempts, exponential backoff (1s, 2s, 4s) for `RetryableError` subclass
- [ ] Non-retryable errors: propagate without retry; task status set to `FAILED` with error JSON
- [ ] Cancellation check: `check_cancellation(encounter_id)` queries Redis flag (set by cancellation event handler)
- [ ] Structured output helper: wraps Vertex AI call with Pydantic schema validation
- [ ] SIGTERM graceful shutdown: inherited by all specialist agents
- [ ] Unit tests: (a) success path, (b) retry on transient error, (c) cancellation check, (d) non-retryable failure
- [ ] Code reviewed and approved

---

## Technical Notes

- `BaseAgent` is a Python abstract base class (ABC); specialist agents override `process(self, event: ADTEvent) -> BaseAgentOutput`
- Cancellation flag stored in Redis key `cancellation:{encounter_id}` with TTL=3600s (set by EP-001 A11/A12/A13 handler)
- `AgentTask` status transitions from base class: `PENDING → IN_PROGRESS` on start, `IN_PROGRESS → COMPLETED/FAILED/CANCELLED` on completion
- Structured output: use LangChain `with_structured_output(PydanticModel)` chain pattern
