# US-020: Build Transition Coordinator Agent with Pub/Sub ADT Subscription

> **Epic:** EP-003 — AI Agent Orchestration Framework | **Sprint:** 2 | **Points:** 8 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** care team member,
**I want to** have the Transition Coordinator Agent automatically receive every ADT event from Pub/Sub and create `AgentTask` records for all 5 specialist agents within 2 seconds,
**So that** the full care transition workflow starts immediately on patient admission, transfer, or discharge without any manual intervention.

---

## Business Value

The coordinator is the heartbeat of SmartHandoff. Its 2-second task creation SLA (FR-004) ensures the 5-second end-to-end ADT-to-notification target (NFR-003) is achievable. Automating task creation eliminates the coordinator overhead that currently costs nurses 45 minutes per transition.

---

## Acceptance Criteria

### Scenario 1: AgentTask records created within 2 seconds of ADT event
**Given** an ADT^A01 event is published to the `adt-events` Pub/Sub topic
**When** the coordinator processes the message
**Then** 5 `AgentTask` records are created in the DB within 2 seconds: `DOCUMENTATION`, `MEDICATION_RECONCILIATION`, `BED_MANAGEMENT`, `FOLLOW_UP_CARE`, `PATIENT_COMMUNICATION`; each with `status=PENDING` and correct `encounter_id`.

### Scenario 2: Coordinator creates only relevant tasks by event type
**Given** an ADT^A02 (transfer) event is received
**When** the coordinator processes it
**Then** only tasks relevant to transfer events are created (e.g. `TRANSFER_NOTE`, `BED_MANAGEMENT`); tasks irrelevant to transfer (e.g. `DISCHARGE_SUMMARY`) are NOT created.

### Scenario 3: Graceful shutdown completes in-flight task before exit
**Given** the coordinator is processing an ADT event and receives a SIGTERM signal
**When** the graceful shutdown handler fires
**Then** the in-flight task creation completes; the Pub/Sub message is acknowledged; the process exits within 30 seconds; no `AgentTask` is left in a partial state.

### Scenario 4: DLQ receives message after 5 failed deliveries
**Given** the coordinator repeatedly fails to process a message (simulated exception)
**When** the 5th delivery attempt fails
**Then** the message is moved to the `adt-events-dlq` subscription; a Cloud Monitoring alert fires; the DLQ message count metric is incremented.

## Definition of Done

- [ ] `TransitionCoordinatorAgent` class with Pub/Sub pull subscription (`asyncio`-based)
- [ ] Task creation logic: event type → task type mapping registered in coordinator config
- [ ] `AgentTask` ORM records created in a single DB transaction for atomicity
- [ ] SIGTERM handler: sets `shutdown_event`; processing loop drains current message and exits
- [ ] Pub/Sub DLQ subscription configured in Terraform (max_delivery_attempts=5)
- [ ] `LangChain` base agent class extracted as a shared library: Pub/Sub consumer, structured output, retry wrapper
- [ ] Performance test: task creation latency p95 <2 seconds under 50 concurrent ADT events
- [ ] Code reviewed and approved

---

## Technical Notes

- Use `google.cloud.pubsub_v1.SubscriberClient.subscribe()` with `FlowControl(max_messages=10)` to prevent memory overload
- SIGTERM handler: `signal.signal(signal.SIGTERM, handler)` sets a `asyncio.Event` that the main loop awaits
- AgentTask creation must be idempotent: use `ON CONFLICT (encounter_id, agent_type) DO NOTHING` to handle redelivered messages
- Pub/Sub message ack deadline: 60 seconds; extend with `modify_ack_deadline` for tasks taking longer than 60s
