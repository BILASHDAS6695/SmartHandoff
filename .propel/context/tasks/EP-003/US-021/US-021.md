# US-021: Implement AgentTask SLA Monitoring & Supervisor Escalation

> **Epic:** EP-003 — AI Agent Orchestration Framework | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Charge Nurse supervisor,
**I want to** be automatically notified when any agent task exceeds its 30-minute SLA threshold,
**So that** I can manually intervene before a care transition is delayed to the point of affecting patient safety.

---

## Business Value

Unacknowledged agent delays directly put patients at risk — a delayed discharge medication reconciliation means a patient could leave without pharmacist review. The 30-minute SLA escalation (BR-014) ensures human oversight catches automation failures before they become patient safety events.

---

## Acceptance Criteria

### Scenario 1: SLA monitor fires escalation at 30-minute threshold
**Given** an `AgentTask` with `agent_type=DOCUMENTATION` was created 30 minutes ago and is still in `IN_PROGRESS` status
**When** the SLA monitor background job runs (every 5 minutes)
**Then** a `SUPERVISOR_ESCALATION` notification message is published to `notification-requests` Pub/Sub topic with: `encounter_id`, `agent_type`, `minutes_elapsed=30`, `supervisor_id`.

### Scenario 2: Task status API returns all tasks for an encounter
**Given** 5 `AgentTask` records exist for encounter `ENC-001`
**When** `GET /api/v1/encounters/ENC-001/tasks` is called with a valid staff JWT
**Then** the response contains all 5 tasks with `id`, `agent_type`, `status`, `start_time`, `completed_time`, `sla_threshold_minutes`, and `sla_breached` flag.

### Scenario 3: Completed tasks do not trigger escalation
**Given** an `AgentTask` with `status=COMPLETED` was created 60 minutes ago
**When** the SLA monitor runs
**Then** no escalation is fired for that task; only `IN_PROGRESS` and `PENDING` tasks past their SLA threshold are evaluated.

### Scenario 4: Configurable SLA threshold per agent type
**Given** `BED_MANAGEMENT` agent has a 15-minute SLA and `DOCUMENTATION` has 30 minutes
**When** the SLA configuration is loaded
**Then** the monitor applies the correct threshold per agent type; `BED_MANAGEMENT` tasks escalate after 15 minutes, not 30.

---

## Definition of Done

- [ ] `SLAMonitor` background task runs every 5 minutes (APScheduler or asyncio `create_task`)
- [ ] SLA thresholds stored as application config (YAML or DB config table) — not hardcoded
- [ ] `sla_breached` boolean flag updated on `AgentTask` record when threshold exceeded
- [ ] Escalation published to Pub/Sub `notification-requests` topic with structured payload
- [ ] `GET /api/v1/encounters/{id}/tasks` endpoint returns all tasks with SLA fields
- [ ] Task status API secured: requires valid staff JWT (EP-011 auth middleware)
- [ ] Unit tests: SLA breach detection logic, config loading, non-escalation of completed tasks
- [ ] Code reviewed and approved

---

## Technical Notes

- SLA monitor uses `READ` DB session (replica routing) to avoid write contention with agent tasks
- Escalation idempotency: only fire one escalation per (encounter_id, agent_type, breach_window) to avoid alert fatigue
- Consider `APScheduler AsyncIOScheduler` for the background job — avoid `time.sleep()` patterns
- Per-agent SLA config: `{DOCUMENTATION: 30, MEDICATION_RECONCILIATION: 60, BED_MANAGEMENT: 15, FOLLOW_UP_CARE: 120, PATIENT_COMMUNICATION: 30}` (minutes)
