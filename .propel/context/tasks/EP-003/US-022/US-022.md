# US-022: Push Real-Time Agent Task Updates via SignalR Hub

> **Epic:** EP-003 — AI Agent Orchestration Framework | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** floor nurse on the care team dashboard,
**I want to** see agent task status changes (PENDING → IN_PROGRESS → COMPLETED) appear on my dashboard within 1 second without refreshing,
**So that** I always have an accurate, live view of care transition progress and can immediately act when a task completes or fails.

---

## Business Value

Real-time visibility (<1s latency, FR-012) is what transforms SmartHandoff from a background automation tool into an active operational command centre. Nurses waiting for discharge summaries or pharmacist alerts can respond immediately rather than polling a page, directly reducing care transition times.

---

## Acceptance Criteria

### Scenario 1: Task status update reaches Angular client within 1 second
**Given** a nurse is viewing the care team dashboard with an active SignalR connection
**When** an agent task transitions from `IN_PROGRESS` to `COMPLETED`
**Then** the dashboard receives a `task_updated` SignalR event within 1 second of the DB write, confirmed by an integration test measuring server-to-client latency.

### Scenario 2: SignalR group routing by encounter, unit, and role
**Given** a nurse subscribed to unit `3A` and Pharmacist Phil subscribed to `medication` role events
**When** a medication reconciliation task completes for an encounter in unit `3A`
**Then** both users receive the `task_updated` event; a nurse in unit `4B` does NOT receive it.

### Scenario 3: SignalR connection drop reconnects automatically
**Given** an Angular SignalR connection drops due to network interruption
**When** the connection is restored
**Then** the Angular client automatically reconnects within 5 seconds; missed task updates since disconnection are re-fetched via `GET /api/v1/encounters/{id}/tasks`.

### Scenario 4: SignalR hub authenticated via JWT
**Given** a SignalR connection attempt is made without a valid JWT
**When** the hub connection is initiated
**Then** the connection is refused with 401 Unauthorized; no group subscriptions are created.

---

## Definition of Done

- [ ] FastAPI SignalR hub implemented: `POST /api/v1/signalr/task-updated` broadcasts to correct groups
- [ ] Group naming convention: `encounter-{id}`, `unit-{unitId}`, `role-{roleName}`
- [ ] Agent task update flow: agent calls SignalR broadcast endpoint after each status transition
- [ ] SignalR hub secured: JWT validation middleware on hub connection
- [ ] Angular `SignalRService` implemented with `@microsoft/signalr` SDK, automatic reconnect strategy
- [ ] Integration test: end-to-end latency from DB write to Angular `task_updated` event ≤1 second
- [ ] Unit tests: group routing logic for encounter/unit/role subscriptions
- [ ] Code reviewed and approved

---

## Technical Notes

- FastAPI does not have native SignalR — use Azure SignalR Service REST API for broadcasting, or self-host with `python-signalr-core` equivalent
- Alternative: Server-Sent Events (SSE) with `fastapi-sse` for simpler implementation if SignalR service latency is acceptable
- Cloud Run supports WebSocket connections; configure `--timeout=3600` for long-lived connections
- Angular service: `HubConnectionBuilder` with `withUrl(url, {accessTokenFactory: () => jwtService.getToken()})` and `withAutomaticReconnect()`
