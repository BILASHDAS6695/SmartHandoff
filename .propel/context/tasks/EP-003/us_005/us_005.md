# US-005: Configure Dead Letter Queues on All Agent Pub/Sub Subscriptions

> **Epic:** EP-003 — AI Agent Orchestration Framework | **Sprint:** 2 | **Points:** 2 | **Priority:** Must Have

## User Story

**As a** DevOps engineer,
**I want** Dead Letter Queues configured on all 6 agent Pub/Sub subscriptions with a Cloud Monitoring alert,
**so that** messages that fail processing 5 times are captured for investigation and never silently lost.

## Acceptance Criteria

- [ ] **AC-1:** All 6 agent subscriptions (`coordinator-sub`, `docs-agent-sub`, `medrecon-sub`, `bed-mgmt-sub`, `followup-sub`, `comms-sub`) configured with `max_delivery_attempts=5` and dead letter topic `adt-events-dlq`
- [ ] **AC-2:** Cloud Monitoring P3 alert fires when `adt-events-dlq` message count >0; alert payload includes subscription name and message count
- [ ] **AC-3:** Agent services implement graceful shutdown: on SIGTERM, in-flight message processing completes (max 30 seconds); unprocessed messages are `nack()`'d so they re-enter the subscription queue
- [ ] **AC-4:** DLQ monitoring dashboard panel shows per-subscription DLQ count over time; zero DLQ messages in normal operation
- [ ] **AC-5:** Integration test: configure a mock agent that always fails → verify message appears in DLQ after 5 delivery attempts → P3 alert fires

## Definition of Done

- [ ] All DLQ configurations verified via `gcloud pubsub subscriptions describe` commands
- [ ] Graceful shutdown tested: SIGTERM sent to agent → in-flight message completes before shutdown

## Dependencies

- EP-TECH US-001 (Pub/Sub topics and subscriptions provisioned)
- EP-003 US-001 (Agent subscription structure established)
