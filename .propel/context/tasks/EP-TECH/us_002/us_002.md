# US-002: Configure Cloud Build CI/CD Pipeline with Canary Deployment

> **Epic:** EP-TECH — Technical Bootstrap & Infrastructure | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have

## User Story

**As a** DevOps engineer,
**I want** an automated Cloud Build pipeline that lints, tests, builds, scans, and deploys with canary traffic splitting on every push to `main`,
**so that** every release is validated automatically and bad deployments roll back before reaching all users.

## Acceptance Criteria

- [ ] **AC-1:** Cloud Build trigger fires on push to `main`; pipeline steps execute in order: lint → unit tests (≥80% coverage gate) → Docker build → Artifact Registry push → vulnerability scan → canary deploy (10% traffic) → 15-minute observation → auto-promote or auto-rollback
- [ ] **AC-2:** Vulnerability scan step rejects images with CRITICAL CVE severity; build fails and deployment is blocked
- [ ] **AC-3:** Canary auto-rollback triggers if API error rate >1% or p95 latency >500ms during 15-minute window
- [ ] **AC-4:** Build notifications (success/failure) sent to team Slack/email webhook
- [ ] **AC-5:** Cloud Deploy delivery pipeline defined for all 10 Cloud Run services with rollout approval gate for `prod` environment

## Definition of Done

- [ ] Pipeline verified end-to-end on a test commit to `main`
- [ ] Canary rollback tested by injecting a deliberate error condition
- [ ] Pipeline configuration in source control (no console-only config)

## Dependencies

- US-001 (Terraform provisions Artifact Registry, Cloud Build, Cloud Deploy)
