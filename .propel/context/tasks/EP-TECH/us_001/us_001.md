# US-001: Provision GCP Infrastructure via Terraform

> **Epic:** EP-TECH — Technical Bootstrap & Infrastructure | **Sprint:** 1 | **Points:** 8 | **Priority:** Must Have

## User Story

**As a** DevOps engineer,
**I want** all GCP resources provisioned via Terraform modules with a single `terraform apply`,
**so that** environments are reproducible, auditable, and require no manual console steps.

## Acceptance Criteria

- [ ] **AC-1:** `terraform apply` from a clean state provisions: Cloud Run services (all 10), Cloud SQL primary + replica, Pub/Sub topics + subscriptions, VPC + subnets + VPC connector, Cloud Memorystore (Redis), Cloud Storage buckets (HL7 archive + audit WORM + ML models), Cloud CDN + Cloud Armor, Secret Manager, Cloud Monitoring workspace
- [ ] **AC-2:** All Cloud Run services deploy with correct CPU, memory, min/max instances, and VPC connector binding as specified in the design (Design §9.2)
- [ ] **AC-3:** Cloud SQL private IP only (no public IP); VPC connector routes Cloud Run traffic to data subnet
- [ ] **AC-4:** `terraform destroy` cleanly removes all resources without manual cleanup
- [ ] **AC-5:** Terraform state stored remotely in GCS backend bucket with versioning enabled
- [ ] **AC-6:** All Terraform modules pass `terraform validate` and `terraform plan` with zero errors in CI

## Definition of Done

- [ ] Code reviewed and merged to `main`
- [ ] `terraform apply` verified in staging project before production
- [ ] No GCP console-provisioned resources in target project
- [ ] Terraform state backend configured and accessible to CI/CD service account

## Dependencies

- GCP project created and billing enabled (prerequisite by GCP Admin)
- Service account with required IAM roles provisioned manually (bootstrap)

## Notes

- Use `us-central1` region for multi-AZ support
- Separate Terraform workspaces for `dev`, `staging`, `prod`
