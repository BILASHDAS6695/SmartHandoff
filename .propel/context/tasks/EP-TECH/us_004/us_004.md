# US-004: Configure GCP Secret Manager for All Service Credentials

> **Epic:** EP-TECH — Technical Bootstrap & Infrastructure | **Sprint:** 1 | **Points:** 2 | **Priority:** Must Have

## User Story

**As a** DevOps engineer,
**I want** all service credentials and API keys stored in GCP Secret Manager and mounted as environment variables in Cloud Run,
**so that** zero secrets are hardcoded in container images, source code, or environment files.

## Acceptance Criteria

- [ ] **AC-1:** All secrets defined in Terraform with placeholder values; populated by SecOps via GCP Console — secrets include: FHIR API credentials, Twilio auth token, SendGrid API key, PHI encryption key, DB password, OIDC client secret, Vertex AI key
- [ ] **AC-2:** Cloud Run services reference secrets via Secret Manager bindings (not env var literals); verified via Cloud Run service YAML showing `secretKeyRef` blocks
- [ ] **AC-3:** No secret values appear in: Cloud Build logs, Cloud Logging entries, container image layers, source code, or Terraform state (state uses secret data source references)
- [ ] **AC-4:** Secret version rotation: new secret version can be deployed by updating Secret Manager without rebuilding containers (Cloud Run picks up new version on next revision)
- [ ] **AC-5:** IAM binding: each Cloud Run service account has `secretmanager.secretAccessor` only for secrets it needs — principle of least privilege verified per service

## Definition of Done

- [ ] Secret Manager bindings verified by deploying a test Cloud Run revision and confirming env var is set
- [ ] Security scan confirms no secret literals in source code (gitleaks or similar)
- [ ] IAM permissions reviewed and documented

## Dependencies

- US-001 (Secret Manager provisioned via Terraform)
