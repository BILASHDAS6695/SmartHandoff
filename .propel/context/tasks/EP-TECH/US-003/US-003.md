# US-003: Implement CI/CD Pipeline with Canary Deployment

> **Epic:** EP-TECH — Technical Bootstrap & Infrastructure | **Sprint:** 1 | **Points:** 8 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** DevOps Engineer,
**I want to** configure a GitHub → Cloud Build CI/CD pipeline that runs lint, test, container build, vulnerability scan, 10% canary deploy, and full promotion on every push to `main`,
**So that** only verified, vulnerability-free code reaches production and rollbacks are automated on failure.

---

## Business Value

Automated canary deployment prevents untested code from impacting all clinical staff simultaneously. Vulnerability scanning in CI (TR-019) satisfies HIPAA security requirements by catching container vulnerabilities before they reach the production environment containing PHI data.

---

## Acceptance Criteria

### Scenario 1: Full pipeline runs on push to main
**Given** a developer pushes a commit to the `main` branch
**When** the GitHub → Cloud Build trigger fires
**Then** the pipeline executes stages in order: lint → unit tests → container build → vulnerability scan → canary deploy (10% traffic) → full promotion (100% traffic), with each stage gating the next on success.

### Scenario 2: Vulnerability scan blocks deployment
**Given** a container image contains a CRITICAL CVE
**When** the vulnerability scan stage runs
**Then** the pipeline fails at the scan stage, no new revision is deployed, and a Slack/email alert is sent to the DevOps team.

### Scenario 3: Canary rollback on error spike
**Given** a canary revision is serving 10% of traffic
**When** the error rate for that revision exceeds 1% within a 5-minute observation window
**Then** the canary is automatically rolled back to the previous revision and the pipeline marks the deployment as failed.

### Scenario 4: Secrets never appear in pipeline logs
**Given** the CI/CD pipeline runs
**When** Cloud Build logs are reviewed
**Then** no plaintext secrets, API keys, or database credentials appear in any log line; all secrets are sourced from Secret Manager at runtime.

---

## Definition of Done

- [ ] Cloud Build trigger configured for push to `main` on all service repositories
- [ ] Pipeline stages: lint (flake8/eslint) → pytest/jest → Docker build → Artifact Registry push → Trivy CVE scan → canary 10% → promote 100%
- [ ] Canary traffic split configured via Cloud Run `--traffic` flag in Cloud Build step
- [ ] Automated rollback on error rate >1% via Cloud Monitoring alert → Cloud Build trigger
- [ ] Pipeline duration <15 minutes for a standard service
- [ ] CRITICAL/HIGH CVEs block deployment; MEDIUM/LOW produce warnings only
- [ ] Code reviewed and approved

---

## Technical Notes

- Use `cloudbuild.yaml` per service; shared steps can be extracted to a `cloudbuild-shared.yaml`
- Trivy image scanning via `aquasec/trivy` action with SARIF output to Cloud Security Command Center
- Cloud Run canary: `gcloud run services update-traffic <svc> --to-revisions=NEW=10,OLD=90`
- Error rate monitoring via Cloud Monitoring custom dashboard + alert policy with auto-resolve action pointing to rollback Cloud Build trigger
- Container images pushed to Artifact Registry (not Docker Hub) for network isolation
