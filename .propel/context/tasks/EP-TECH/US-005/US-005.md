# US-005: Enforce Zero-Hardcoded-Secrets via Secret Manager

> **Epic:** EP-TECH — Technical Bootstrap & Infrastructure | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Security Engineer,
**I want to** ensure all application secrets (database credentials, API keys, FHIR client secrets, Twilio/SendGrid tokens) are stored in GCP Secret Manager and mounted at runtime — never hardcoded in source code or container images,
**So that** the HIPAA requirement of zero exposed credentials is met and secrets can be rotated without redeployment.

---

## Business Value

Hardcoded secrets in source code are the most common cause of healthcare data breaches. This story directly enforces TR-021 and prevents HIPAA violations that carry penalties exceeding $1.9M per violation category. Secret Manager versioning also enables zero-downtime credential rotation.

---

## Acceptance Criteria

### Scenario 1: Secret Manager holds all required secrets
**Given** the Terraform `secrets` module is applied
**When** `gcloud secrets list --project={project}` is executed
**Then** all required secret names exist (DB password, FHIR client secret, Twilio auth token, SendGrid API key, JWT signing key, OIDC client secret, GCS HMAC key) with at least one version in ENABLED state.

### Scenario 2: Container images contain no secrets
**Given** a production container image in Artifact Registry
**When** the image layers are scanned by `trufflehog` or `gitleaks`
**Then** zero secrets, API keys, or credential patterns are found in any image layer.

### Scenario 3: Secret rotation does not require redeployment
**Given** a database password secret is rotated in Secret Manager
**When** the new version is set to ENABLED and the old version DISABLED
**Then** Cloud Run services reading that secret pick up the new value within 60 seconds of the next request (or on the next Cloud Run instance start) without a manual redeployment.

### Scenario 4: Pre-commit hook blocks secret commits
**Given** `gitleaks` pre-commit hook is installed
**When** a developer attempts to commit a file containing an AWS access key pattern or a database URL with credentials
**Then** the commit is blocked with a descriptive error message listing the detected secret pattern and file location.

---

## Definition of Done

- [ ] All Secret Manager secrets declared in the Terraform `secrets` module with placeholder values
- [ ] Cloud Run services use `secretenv` or volume mounts to access secrets at runtime
- [ ] `gitleaks` pre-commit hook added to `.pre-commit-config.yaml` with project-appropriate rulesets
- [ ] Secret scanning step added to CI/CD pipeline (scans both source code and built image)
- [ ] IAM: each Cloud Run service account has `secretmanager.secretAccessor` only for its own secrets (principle of least privilege)
- [ ] Secret rotation procedure documented in `infra/BOOTSTRAP.md`
- [ ] Code reviewed and approved by Security Engineer

---

## Technical Notes

- Secret Manager automatic rotation with Cloud Functions trigger is out of scope for Sprint 1; manual rotation procedure is acceptable
- Use `secretmanager.SecretAccessor` role on individual secret resources, NOT at the project level
- For CMEK: Cloud SQL uses Cloud KMS key managed in the `secrets` module
- Terraform `google_secret_manager_secret_version` should use `sensitive = true` for the `secret_data` attribute to prevent logging in plan output
