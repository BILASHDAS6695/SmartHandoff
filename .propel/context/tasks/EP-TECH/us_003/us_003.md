# US-003: Configure Cloud Monitoring Dashboards and Alerting

> **Epic:** EP-TECH — Technical Bootstrap & Infrastructure | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have

## User Story

**As a** DevOps engineer,
**I want** Cloud Monitoring dashboards and alerting policies configured for all services,
**so that** the team has real-time operational visibility and is paged before users notice incidents.

## Acceptance Criteria

- [ ] **AC-1:** Dashboard shows per-service metrics: request count, error rate, p50/p95/p99 latency, CPU, memory, instance count — for all 10 Cloud Run services
- [ ] **AC-2:** P1 alert fires within 2 minutes when API error rate >1% over a 5-minute window; notification sent to on-call channel
- [ ] **AC-3:** P1 alert fires when ADT event Pub/Sub processing lag >10 seconds
- [ ] **AC-4:** P1 alert fires when Cloud SQL replication lag >30 seconds
- [ ] **AC-5:** P2 alert fires when agent task failure rate >5%
- [ ] **AC-6:** P3 alert fires when any DLQ message count >0
- [ ] **AC-7:** 99.9% uptime check configured for `GET /health` endpoint on API Gateway; alert on 2 consecutive failures
- [ ] **AC-8:** OpenTelemetry traces visible in Cloud Trace for end-to-end request flows; `X-Cloud-Trace-Context` header propagated across all services

## Definition of Done

- [ ] All alert policies terraform-managed (not console-created)
- [ ] On-call rotation notification channel configured and tested with a manual alert trigger
- [ ] Trace sampling verified via test request through full ADT event pipeline

## Dependencies

- US-001 (Cloud Monitoring workspace provisioned via Terraform)
