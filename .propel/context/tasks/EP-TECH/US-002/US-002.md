# US-002: Configure Cloud Run Service Manifests with Health Probes

> **Epic:** EP-TECH — Technical Bootstrap & Infrastructure | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** DevOps Engineer,
**I want to** define Cloud Run service manifests for all 10 services with correct CPU/memory allocations, min/max instances, VPC connector bindings, and liveness/readiness probes,
**So that** services scale reliably under load, unhealthy instances are replaced automatically, and the system meets the 99.9% uptime SLA.

---

## Business Value

Correct Cloud Run configuration directly prevents overloading under peak ADT load (5,000 events/day) and ensures zero-downtime deployments. Health probes prevent traffic routing to cold or broken instances, protecting clinical staff from dashboard outages during critical care transitions.

---

## Acceptance Criteria

### Scenario 1: All services deploy with correct resource specs
**Given** the Cloud Run Terraform module is applied
**When** `gcloud run services describe <service>` is called for each of the 10 services
**Then** CPU, memory, min-instances, and max-instances match the values in the Terraform `variables.tf` and no service uses the Cloud Run default.

### Scenario 2: Readiness probe prevents premature traffic
**Given** a new Cloud Run revision is deploying
**When** the readiness probe endpoint (`GET /ready`) returns 503 during startup
**Then** traffic is not routed to that revision until the probe returns 200.

### Scenario 3: Liveness probe triggers container restart
**Given** a Cloud Run instance's application process has deadlocked
**When** the liveness probe (`GET /health`) fails 3 consecutive times
**Then** Cloud Run restarts the container and logs the event to Cloud Logging.

### Scenario 4: VPC connector binding is active
**Given** a Cloud Run service that requires database access
**When** the service attempts a connection to the Cloud SQL private IP
**Then** the connection succeeds via the VPC connector; direct internet path is rejected.

---

## Definition of Done

- [ ] Terraform `cloud_run` module parameterises CPU, memory, min/max instances, concurrency, and VPC connector per service
- [ ] All 10 service definitions have `liveness_probe` and `readiness_probe` configured
- [ ] `GET /health` returns `{"status": "ok"}` for all services in the dev environment
- [ ] `GET /ready` returns `{"status": "ready"}` only after startup dependencies (DB, Redis) are reachable
- [ ] Multi-zone deployment verified (services spread across ≥2 GCP zones)
- [ ] Code reviewed and approved

---

## Technical Notes

- MLLP listener service (EP-001) does not use HTTP probes — use TCP probe on port 2575
- Service-to-service communication uses Cloud Run service accounts and Workload Identity
- Minimum instances for critical services (coordinator, MLLP): min=1 to avoid cold-start latency on ADT events
- `startup_probe` recommended for services with slow framework initialisation (LangChain agents)
