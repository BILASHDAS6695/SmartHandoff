# US-001: Provision GCP Infrastructure via Terraform IaC

> **Epic:** EP-TECH — Technical Bootstrap & Infrastructure | **Sprint:** 1 | **Points:** 8 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** DevOps Engineer,
**I want to** provision all GCP resources (Cloud Run, Cloud SQL, Pub/Sub, VPC, Cloud Armor, Cloud Memorystore, Secret Manager) with a single `terraform apply`,
**So that** the entire infrastructure is reproducible, version-controlled, and deployed without any manual console steps.

---

## Business Value

A fully automated IaC foundation eliminates configuration drift and manual provisioning errors, enabling every downstream epic to begin work on a stable, compliant GCP environment. This directly satisfies TR-018 (100% IaC) and NFR-022 (RTO <1 hour) because destroyed environments can be rebuilt automatically.

---

## Acceptance Criteria

### Scenario 1: Clean-slate provisioning succeeds
**Given** an empty GCP project with the Terraform state backend configured
**When** `terraform apply` is executed for the dev environment
**Then** all resources (Cloud Run services, Cloud SQL HA instance, Pub/Sub topics, VPC, Cloud Armor policy, Redis instance, Secret Manager secrets) are created with zero manual console steps and `terraform plan` shows no delta immediately after apply.

### Scenario 2: VPC isolation enforced for data tier
**Given** the networking module has been applied
**When** a direct internet connection attempt is made to the Cloud SQL private IP
**Then** the connection is refused; only Cloud Run instances with the VPC connector can reach Cloud SQL.

### Scenario 3: Cloud SQL HA configuration active
**Given** the Cloud SQL module has been applied
**When** a zone-level failure is simulated
**Then** the HA replica promotes within 60 seconds and PITR is enabled with automated backups configured every 4 hours.

### Scenario 4: No secrets in source code
**Given** the Terraform modules are applied
**When** every Cloud Run service's environment variables and container image layers are inspected
**Then** zero plaintext secrets are found; all secrets are mounted from Secret Manager.

---

## Definition of Done

- [ ] Terraform modules authored for: `cloud_run`, `cloud_sql`, `networking`, `pubsub`, `redis`, `secrets`, `storage`, `armor_lb_cdn`
- [ ] `terraform apply` on dev environment completes with zero errors
- [ ] `terraform plan` post-apply shows no changes (idempotent)
- [ ] All Cloud SQL instances have HA replica, PITR enabled, and automated backups every 4 hours
- [ ] VPC connector configured; Cloud SQL has no public IP
- [ ] Secret Manager contains placeholder secrets for all services
- [ ] Code reviewed and approved by senior DevOps engineer
- [ ] No hardcoded credentials in any `.tf`, `.tfvars`, or CI configuration files

---

## Technical Notes

- Use `terraform workspaces` or environment-specific directories (`environments/dev`, `environments/staging`, `environments/prod`)
- Cloud SQL must use `CMEK` (customer-managed encryption key) via Cloud KMS
- VPC Connector required for all Cloud Run services needing DB/Redis access
- Cloud Armor WAF policy must attach to the external HTTPS load balancer at module level
- Pub/Sub topics must include DLQ subscriptions configured at provisioning time (max_delivery_attempts=5)
