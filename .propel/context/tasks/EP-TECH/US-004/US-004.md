# US-004: Set Up Cloud Monitoring Dashboards & Alerting

> **Epic:** EP-TECH — Technical Bootstrap & Infrastructure | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** DevOps Engineer,
**I want to** configure Cloud Monitoring dashboards, P1/P2/P3 alerting policies, uptime checks, and OpenTelemetry distributed tracing,
**So that** operations teams can detect and respond to service degradation before it impacts clinical workflows.

---

## Business Value

Proactive observability is critical for a healthcare system with a 99.9% uptime SLA (NFR-020) and RTO <1 hour (NFR-022). Distributed tracing across all agents enables rapid diagnosis of slow ADT-to-notification paths, directly protecting the <5-second end-to-end SLA (FR-001).

---

## Acceptance Criteria

### Scenario 1: P1 alert fires on error rate spike
**Given** a Cloud Monitoring P1 alert policy is configured for error rate >1%
**When** a service emits error logs at a rate exceeding 1% of requests for 60 seconds
**Then** a PagerDuty/email notification fires within 2 minutes and the Cloud Monitoring dashboard shows the alert as active.

### Scenario 2: Distributed trace visible end-to-end
**Given** OpenTelemetry is configured across all 10 services
**When** an HL7 ADT A01 event is processed end-to-end (MLLP → coordinator → documentation agent)
**Then** a single trace in Cloud Trace shows every span from MLLP receipt to SignalR push with correct parent-child relationships and latency breakdown.

### Scenario 3: Uptime check detects service down
**Given** uptime checks are configured for all public endpoints
**When** a service's `GET /health` endpoint returns non-2xx for 2 consecutive checks (60-second interval)
**Then** an uptime alert fires and the affected service shows as "failing" on the Cloud Monitoring dashboard.

### Scenario 4: PHI redaction in structured logs
**Given** the PHI redaction log sink is active
**When** a service logs a message containing a patient name or MRN
**Then** the Cloud Logging export shows those fields replaced with `[REDACTED]`; raw logs in the secure sink retain the original values with restricted IAM access.

---

## Definition of Done

- [ ] Cloud Monitoring dashboards created: service health, ADT throughput, agent latency, error rates
- [ ] P1 alert: error rate >1% for 60s; P2: latency p95 >5s; P3: DLQ message count >0
- [ ] Uptime checks configured for all 10 service `/health` endpoints (60s interval)
- [ ] OpenTelemetry SDK integrated into each service; traces visible in Cloud Trace
- [ ] Structured logging format (JSON) with PHI redaction middleware active in all services
- [ ] Alert notification channels configured (email + PagerDuty integration)
- [ ] Code reviewed and approved

---

## Technical Notes

- Use `opentelemetry-sdk` Python package with `google-cloud-trace` exporter
- PHI redaction: use Cloud Logging exclusion filters + log sink with Dataflow transform for secure audit copy
- Alert policies defined as Terraform resources in the `monitoring` module to maintain IaC compliance
- Dashboard JSON exported as Terraform `google_monitoring_dashboard` resource
