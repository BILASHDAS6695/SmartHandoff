# US-037: Score and Recommend Optimal Bed Assignment for Incoming Patients

> **Epic:** EP-006 — Bed Management Agent | **Sprint:** 2 | **Points:** 3 | **Priority:** Should Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** bed manager,
**I want to** receive a ranked list of ≥3 recommended beds for an incoming patient based on their acuity, care type, isolation status, and gender designation,
**So that** I can select the optimal bed placement in seconds rather than manually reviewing every unit's availability.

---

## Business Value

Manual bed assignment by phone calls takes 30–45 minutes on average. Algorithmic scoring reduces this to a 30-second decision. Better matching of patient acuity to bed type also reduces inappropriate placements that result in secondary transfers (BO-05 contribution).

---

## Acceptance Criteria

### Scenario 1: Recommendation returns ≥3 ranked beds
**Given** a pending A01 admit for a patient requiring `ICU-step-down`, no isolation, `female` designation
**When** `GET /api/v1/beds/recommend?encounter_id={id}` is called
**Then** the response contains ≥3 available beds ranked by score, each with `bed_id`, `unit`, `room`, `score`, and `score_breakdown` (acuity_match, care_type_match, isolation_available, gender_match).

### Scenario 2: Isolation patients only recommended to isolation-capable beds
**Given** a patient requires airborne isolation (tuberculosis)
**When** the recommendation algorithm runs
**Then** only beds in isolation-capable rooms are included in the results; non-isolation beds receive a score of 0 and are excluded.

### Scenario 3: Bed score algorithm uses configurable weights
**Given** the weight configuration: `acuity=0.4, care_type=0.35, isolation=0.15, gender=0.10`
**When** a bed is scored
**Then** the `score` field equals the weighted sum of each factor (0–1 scale); the weights sum to 1.0.

### Scenario 4: No available beds returns empty results with advisory
**Given** all beds in the patient's required unit are OCCUPIED or DIRTY
**When** the recommendation API is called
**Then** an empty `recommendations=[]` is returned with `advisory={"message": "No beds available in requested unit. Nearest available unit: [unit_name]", "available_unit": "...", "estimated_wait_minutes": N}`.

## Definition of Done

- [ ] `BedScoringAlgorithm` class with configurable weight YAML
- [ ] Scoring factors: `acuity_match`, `care_type_match`, `isolation_match`, `gender_match` (each 0–1)
- [ ] `GET /api/v1/beds/recommend` endpoint: takes `encounter_id`; fetches encounter + VACANT beds; scores each; returns top 5
- [ ] Recommendation result includes `score_breakdown` for transparency
- [ ] No-beds scenario: advisory with nearest available unit and estimated wait
- [ ] Unit tests: scoring weights, isolation filter, no-beds advisory
- [ ] Code reviewed and approved

---

## Technical Notes

- Query VACANT beds from `mv_bed_board` (replica) for sub-500ms response
- Encounter features: `admit_type`, `isolation_required`, `acuity_level`, `gender` from ADTEvent
- Weight config: `config/bed_scoring_weights.yaml` — hot-reloadable without deployment
- Score range: 0.0–1.0; higher is better match; sort descending, return top 5
