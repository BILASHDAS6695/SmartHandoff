# US-035: Maintain Real-Time Bed Availability Board

> **Epic:** EP-006 — Bed Management Agent | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** bed manager (Coordinator Carol),
**I want to** see an automatically updated bed availability board that reflects the current status (OCCUPIED, VACANT, DIRTY, RESERVED) of every bed within 60 seconds of an ADT event,
**So that** I can coordinate patient placement in real time without making phone calls to nursing units to ask about bed availability.

---

## Business Value

Real-time bed visibility directly reduces ED boarding time from 4+ hours to <2 hours (BO-05, 40% reduction). A bed board that is minutes out-of-date causes placement errors, patient safety incidents, and ED overcrowding. The 60-second update SLA (DR-007) matches the materialised view refresh cadence.

---

## Acceptance Criteria

### Scenario 1: Bed status updated within 60 seconds of A01 (admit)
**Given** bed `3A-02` is VACANT
**When** an A01 ADT event assigns a patient to that bed
**Then** the bed board shows bed `3A-02` as `OCCUPIED` within 60 seconds; the `mv_bed_board` materialised view is refreshed.

### Scenario 2: Bed set to DIRTY on A03 discharge
**Given** bed `4B-01` is OCCUPIED
**When** an A03 discharge event occurs for the patient in that bed
**Then** bed `4B-01` transitions to `DIRTY` (awaiting housekeeping); housekeeping notification is published within 5 seconds.

### Scenario 3: Bed board API returns filtered results with correct status
**Given** Unit `3A` has 10 beds: 5 OCCUPIED, 3 DIRTY, 2 VACANT
**When** `GET /api/v1/beds?unit=3A&status=VACANT` is called
**Then** exactly 2 beds are returned with correct bed IDs and status; response p95 <500ms.

### Scenario 4: Bed inventory seeded from configuration on first deploy
**Given** `config/bed_inventory.yaml` defines 200 beds across 5 units
**When** the bed management service starts for the first time
**Then** 200 `bed` records are created in the DB; the `mv_bed_board` materialised view is populated; no duplicates are created on service restart (idempotent seeding).

## Definition of Done

- [ ] `BedManagementAgent` extends `BaseAgent`; processes A01 (OCCUPIED), A02 (dirty+occupied), A03 (DIRTY) events
- [ ] `bed` ORM table with: `unit`, `room`, `bed_number`, `bed_type`, `status`, `isolation_required`, `gender_designation`
- [ ] `mv_bed_board` materialised view refresh triggered after each bed status change
- [ ] Bed inventory seeding on startup: idempotent `INSERT ... ON CONFLICT DO NOTHING` from YAML config
- [ ] Housekeeping Pub/Sub notification: `notification-requests` topic within 5 seconds of A03
- [ ] `GET /api/v1/beds` endpoint with `unit`, `status`, `bed_type` query filters; routed to read replica
- [ ] `PATCH /api/v1/beds/{id}/status` endpoint for manual bed status override (bed manager role)
- [ ] Unit tests: A01/A02/A03 status transitions, bed board filter API
- [ ] Code reviewed and approved

---

## Technical Notes

- Materialised view trigger: after `UPDATE bed SET status=...`, trigger `REFRESH MATERIALIZED VIEW CONCURRENTLY mv_bed_board` in a background async task
- `CONCURRENTLY` refresh avoids locking the view during refresh — requires a unique index on the view
- Bed status enum: `VACANT`, `OCCUPIED`, `DIRTY`, `MAINTENANCE`, `RESERVED` (configurable)
- Bed board API: query `mv_bed_board` (replica) not the base `bed` table directly for performance
