# US-036: Predict Patient Discharge Time with ML Model

> **Epic:** EP-006 — Bed Management Agent | **Sprint:** 2 | **Points:** 5 | **Priority:** Should Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** bed manager (Coordinator Carol),
**I want to** see an ML-predicted discharge window (±2 hours) for each occupied bed on the bed board,
**So that** I can proactively reserve receiving beds and start housekeeping preparation 2 hours before projected discharge, reducing bed turnaround time.

---

## Business Value

Proactive bed preparation based on discharge predictions reduces the time between discharge and a new patient assignment — a key driver of ED boarding time. A ±2-hour prediction window (FR-040) is operationally useful: it enables preparation without creating false certainty that delays real-time response.

---

## Acceptance Criteria

### Scenario 1: ML inference returns prediction within 500ms
**Given** the ML Inference Service is deployed with the trained Scikit-learn model
**When** `POST /ml-inference/predict/discharge-time` is called with encounter features
**Then** a `predicted_discharge_time` (ISO datetime) and `confidence_interval` (±hours) are returned within 500ms (TR-007).

### Scenario 2: Prediction window is ±2 hours on holdout test set
**Given** the model is evaluated on the 20% holdout dataset
**When** predictions are compared to actual discharge times
**Then** 80% of predictions fall within ±2 hours of actual discharge; MAE ≤2 hours.

### Scenario 3: Prediction updates when patient status changes
**Given** a patient's acuity score changes after lab results
**When** the bed management agent recomputes features and calls the inference service
**Then** the `predicted_discharge_time` in the bed board is updated within 60 seconds of the status change.

### Scenario 4: Prediction displayed on bed board with confidence indicator
**Given** the bed board renders bed `3A-01` with `predicted_discharge_time` in 3 hours
**When** a bed manager views the bed board
**Then** the bed entry shows the predicted discharge time and a colour-coded confidence indicator (green: high confidence, yellow: medium, red: low/unknown).

## Definition of Done

- [ ] ML Inference Service Cloud Run: serves `GradientBoostingRegressor` model via `joblib`
- [ ] Model training pipeline: Jupyter/Python notebook with feature engineering, training, and evaluation steps
- [ ] Features: `admit_diagnosis_group`, `patient_age`, `los_so_far_hours`, `pending_procedures`, `unit`, `day_of_week`
- [ ] Model evaluation: MAE, RMSE, % within ±2 hours on holdout set (threshold: ≥80%)
- [ ] `POST /ml-inference/predict/discharge-time` FastAPI endpoint (auth: service account JWT)
- [ ] Prediction stored in `encounter.predicted_discharge_time` and reflected in `mv_bed_board`
- [ ] Model versioning: model stored in GCS with version tag; inference service loads latest model on startup
- [ ] Unit tests: inference endpoint, feature vector construction
- [ ] Code reviewed and approved

---

## Technical Notes

- Model file: `models/discharge_time_v1.joblib` stored in GCS `ml-models` bucket
- Feature engineering: `los_so_far_hours = (now - encounter.admit_time).total_seconds() / 3600`
- Model refresh: retrain nightly with new encounter data; new model version deployed via CI/CD
- Confidence indicator thresholds: `high if std_dev < 1hr, medium if 1-2hr, low if >2hr`
