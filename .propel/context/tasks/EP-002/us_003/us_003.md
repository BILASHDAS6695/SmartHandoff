# US-003: Enforce FHIR Data Transience — No PHI Written to SmartHandoff DB

> **Epic:** EP-002 — EHR / FHIR Integration | **Sprint:** 1 | **Points:** 2 | **Priority:** Must Have

## User Story

**As a** compliance officer,
**I want** FHIR-fetched patient data to remain in agent memory only during task execution and never written to the SmartHandoff database,
**so that** PHI is not duplicated across systems and the EHR remains the single source of truth (per Constraint C-03).

## Acceptance Criteria

- [ ] **AC-1:** FHIR client returns in-memory `FHIRPatient`, `FHIRMedication`, etc. objects; no ORM `session.add()` call is made for FHIR resource data — enforced by a linting rule or abstract base class that raises `FHIRDataPersistenceError` if attempted
- [ ] **AC-2:** After agent task completion, FHIR objects go out of scope and are garbage-collected; no FHIR PHI fields (name, DOB, MRN in plaintext) appear in SmartHandoff DB tables other than the encrypted columns in `patient` (which are populated from HL7, not FHIR)
- [ ] **AC-3:** Integration test: run full medication reconciliation agent task → query SmartHandoff DB → confirm FHIR `MedicationStatement` raw resource data is absent from all tables; only processed `Medication` records (with mapped fields) are present
- [ ] **AC-4:** Architecture Decision ADR-007 documented in codebase README confirming FHIR transience rationale and enforcement mechanism

## Definition of Done

- [ ] Linting rule or type system constraint prevents accidental FHIR-to-DB persistence (verified in CI)
- [ ] DB inspection test passes confirming no raw FHIR JSON in any column after agent task

## Dependencies

- EP-002 US-001 (FHIR client defined)
- EP-DATA US-001 (DB schema defined — used as reference for what IS permitted)
