# US-016: Implement SMART on FHIR OAuth 2.0 Client with Token Cache

> **Epic:** EP-002 — EHR / FHIR Integration | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Backend Engineer,
**I want to** implement a SMART on FHIR OAuth 2.0 client credentials flow that authenticates with the EHR, caches the access token with a 60-second expiry buffer, and auto-refreshes before expiry,
**So that** all agents can fetch FHIR resources without re-authenticating on every request, reducing EHR API load and latency.

---

## Business Value

SMART on FHIR authentication is the gateway to all patient data. Token caching prevents the N×100 FHIR requests/day from each triggering a separate OAuth round-trip, reducing EHR auth server load. The 60-second refresh buffer prevents token expiry mid-request, which would cause agent task failures during critical care transitions.

---

## Acceptance Criteria

### Scenario 1: Client credentials flow authenticates successfully
**Given** valid FHIR client credentials in Secret Manager
**When** `FHIRClient.authenticate()` is called
**Then** an access token is obtained from the EHR's SMART on FHIR token endpoint using `client_credentials` grant type; the token is cached in memory with expiry = `expires_in - 60` seconds.

### Scenario 2: Cached token is reused without re-authentication
**Given** a valid cached token with 120 seconds remaining
**When** `FHIRClient.get_access_token()` is called
**Then** the cached token is returned immediately; no network request is made to the auth server (verified via mock: auth endpoint not called).

### Scenario 3: Token refreshed before expiry buffer
**Given** a cached token with 55 seconds remaining (within the 60-second buffer)
**When** `FHIRClient.get_access_token()` is called
**Then** a new token request is sent to the auth server; the new token replaces the cached value; the returned token has a fresh expiry.

### Scenario 4: Authentication failure is logged and raises a circuit-breaker-compatible exception
**Given** the EHR auth server returns a 401 Unauthorized
**When** `FHIRClient.authenticate()` is called
**Then** a `FHIRAuthenticationError` is raised with the HTTP status and response body; the error is logged as CRITICAL to Cloud Logging with no PHI in the log message.

---

## Definition of Done

- [ ] `FHIRAuthClient` class implemented with `httpx.AsyncClient` for async requests
- [ ] OAuth 2.0 `client_credentials` grant POST to SMART on FHIR token endpoint
- [ ] In-memory token cache: `TokenCache` dataclass with `access_token` and `expires_at` fields
- [ ] Token cache thread-safe via `asyncio.Lock` (prevents simultaneous refresh race)
- [ ] Client credentials loaded from Secret Manager (not environment variables)
- [ ] Unit tests: (a) successful auth with mock server, (b) cache hit, (c) cache miss triggers refresh, (d) 401 raises `FHIRAuthenticationError`
- [ ] Code reviewed and approved

---

## Technical Notes

- SMART on FHIR well-known discovery: `GET {base_url}/.well-known/smart-configuration` to discover token endpoint
- Use `httpx.AsyncClient` with a shared connection pool (reuse across requests)
- Scope: `system/*.read` (or per-resource scopes as required by the EHR)
- Credentials stored as separate secrets: `fhir_client_id` and `fhir_client_secret` in Secret Manager
