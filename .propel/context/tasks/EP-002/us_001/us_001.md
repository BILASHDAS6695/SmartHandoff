# US-001: Implement FHIR R4 Client with SMART on FHIR Authentication

> **Epic:** EP-002 — EHR / FHIR Integration | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have

## User Story

**As a** backend engineer,
**I want** a reusable async FHIR R4 client library that authenticates via OAuth 2.0 SMART on FHIR client credentials,
**so that** all agents can securely fetch patient data from the EHR with a single shared, tested client.

## Acceptance Criteria

- [ ] **AC-1:** FHIR client uses `httpx.AsyncClient`; authenticates via SMART on FHIR client credentials flow; access token cached in memory with 60-second expiry buffer and auto-refresh
- [ ] **AC-2:** Client credentials (`client_id`, `client_secret`, FHIR base URL) loaded from GCP Secret Manager via environment variable reference — no hardcoded values
- [ ] **AC-3:** Typed Pydantic wrapper models defined for 7 FHIR R4 resources: `Patient`, `Encounter`, `MedicationStatement`, `MedicationAdministration`, `MedicationRequest`, `AllergyIntolerance`, `Condition` — validated against R4 schemas on ingest
- [ ] **AC-4:** `GET Patient?identifier={system}|{mrn}` returns a typed `FHIRPatient` model; missing required fields raise `FHIRValidationError`
- [ ] **AC-5:** Rate limiter: token bucket limiting to 100 FHIR requests/minute per client instance; requests exceeding limit are delayed (not dropped) with exponential backoff
- [ ] **AC-6:** Unit tests mock FHIR responses for all 7 resource types; 100% branch coverage on auth token refresh logic

## Definition of Done

- [ ] Client library published as internal Python package importable by all agent services
- [ ] Integration test against a FHIR R4 sandbox (HAPI FHIR test server) verifies all 7 resource fetches
- [ ] Token refresh race condition tested (concurrent requests during token expiry window)
