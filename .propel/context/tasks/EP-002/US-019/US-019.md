# US-019: Resolve Patient Identity from FHIR via MRN with Partial-Match Fallback

## User Story

**As a** Backend Engineer,
**I want to** resolve a patient's FHIR identity using MRN as the primary lookup, falling back to name + DOB when MRN resolution fails, and creating a partial encounter record when the patient cannot be fully resolved,
**So that** agents can always proceed with the best available patient data rather than failing completely for unresolvable patients.

---

## Business Value

Not all EHR patients have clean MRN data â€” edge cases include unregistered trauma patients and visiting patients. Graceful partial resolution ensures agents can still generate preliminary documentation with available data, preventing total workflow failure for edge-case patients.

---

## Acceptance Criteria

### Scenario 1: Patient resolved by MRN on first attempt
**Given** a patient with MRN `MRN-789` exists in FHIR with `identifier.type.coding.code = MR`
**When** `resolve_patient(mrn="MRN-789")` is called
**Then** the full `PatientModel` is returned with `resolution_method=MRN`; no fallback query is executed.

### Scenario 2: Patient resolved by name + DOB fallback
**Given** no patient with MRN `MRN-789` is found in FHIR
**When** the fallback query `Patient?family=Smith&birthdate=1980-01-15` is attempted
**Then** if exactly one matching patient is returned, the `PatientModel` is returned with `resolution_method=NAME_DOB, partial_match=True`; a warning is logged.

### Scenario 3: Multiple name+DOB matches produces ambiguous result
**Given** the name + DOB fallback returns 3 matching patients
**When** the resolver processes the response
**Then** a `PatientAmbiguousError` is raised; the encounter is created with `patient_resolution_status=AMBIGUOUS`; a care team alert is created for manual resolution; the AI agents are paused for that encounter.

### Scenario 4: Unresolvable patient creates partial encounter
**Given** both MRN and name+DOB lookups return zero results
**When** the encounter creation is attempted
**Then** the encounter is created with `patient_resolution_status=UNRESOLVED`; agent tasks are created with `status=BLOCKED`; a `PatientNotFoundWarning` is logged with the encounter ID for clinical staff follow-up.

## Technical Notes

- MRN system URI must be configurable (different hospitals use different FHIR identifier system URIs)
- Store `patient_resolution_status` in the `encounter` table (not the patient table) â€” the patient record itself may be correct
- Care team alert for ambiguous/unresolved patients dispatched via Pub/Sub `notification-requests` (same as EP-013 infra)
