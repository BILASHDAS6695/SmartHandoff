# US-018: Implement FHIR Client Circuit Breaker & Exponential Backoff Retry

> **Epic:** EP-002 — EHR / FHIR Integration | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Backend Engineer,
**I want to** add exponential backoff retry (3 attempts: 1s/2s/4s) and a circuit breaker (open after 10 failures in 60s, half-open probe after 120s) to all FHIR API calls,
**So that** transient EHR outages are survived transparently and a degraded EHR is not hammered with requests that would worsen recovery time.

---

## Business Value

EHR systems undergo maintenance windows and transient failures. Without retry, a single network blip causes agent task failures and missing clinical data. The circuit breaker prevents SmartHandoff from contributing to an EHR outage by throttling requests — protecting patient safety during periods when the EHR is degraded.

---

## Acceptance Criteria

### Scenario 1: Retry succeeds after one transient failure
**Given** the FHIR server returns HTTP 503 on the first attempt
**When** `FHIRClient.get_patient_by_mrn()` is called
**Then** the client retries after 1 second; the second attempt (HTTP 200) succeeds and the `PatientModel` is returned; the retry is logged as a warning.

### Scenario 2: Circuit breaker opens after 10 consecutive failures
**Given** the FHIR server has been returning HTTP 503 for all requests
**When** the 10th consecutive failure occurs within 60 seconds
**Then** the circuit breaker state transitions to `OPEN`; subsequent calls immediately raise `FHIRCircuitOpenError` without making an HTTP request; a CRITICAL alert is logged.

### Scenario 3: Circuit breaker probes after 120 seconds
**Given** the circuit breaker is in `OPEN` state for 120 seconds
**When** the next call is made
**Then** one probe request is sent to the FHIR server; if it returns HTTP 200, the circuit transitions to `CLOSED`; if it fails, it remains `OPEN` for another 120 seconds.

### Scenario 4: Rate limiter enforces 100 requests/min per instance
**Given** an agent instance is making FHIR requests
**When** the 101st FHIR request within 60 seconds is attempted
**Then** the token bucket rate limiter delays the request (not drops it) until a token is available; the delay is logged at DEBUG level.

---

## Definition of Done

- [ ] `@retry(max_attempts=3, backoff=[1, 2, 4])` decorator applied to all FHIR fetch methods
- [ ] Circuit breaker implemented using `pybreaker` or custom `asyncio`-compatible implementation
- [ ] Circuit breaker thresholds: 10 failures / 60s window → OPEN; 120s reset timeout
- [ ] Circuit breaker state persisted in module-level singleton (per Cloud Run instance)
- [ ] Token bucket rate limiter: 100 tokens/minute refill, applied globally per FHIR client instance
- [ ] Unit tests: (a) retry on transient error, (b) circuit open after threshold, (c) circuit probe logic, (d) rate limiter delay
- [ ] Prometheus metrics: `fhir_circuit_state`, `fhir_retry_total`, `fhir_rate_limited_total`
- [ ] Code reviewed and approved

---

## Technical Notes

- `pybreaker` library is synchronous; wrap with `asyncio.get_event_loop().run_in_executor` or use an async-native circuit breaker
- Token bucket: `asyncio.Semaphore` with periodic release task works for simple cases; use `aiolimiter` for production-grade token bucket
- Retry should NOT retry on 4xx responses (client errors) — only on 5xx, network timeouts, and connection errors
- Circuit breaker metrics should be exported to Cloud Monitoring as custom metrics
