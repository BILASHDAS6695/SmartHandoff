# US-017: Fetch & Validate FHIR R4 Resources with Typed Pydantic Models

> **Epic:** EP-002 — EHR / FHIR Integration | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Backend Engineer,
**I want to** build async FHIR R4 fetch methods for Patient, Encounter, MedicationStatement, MedicationAdministration, MedicationRequest, AllergyIntolerance, and Condition resources, validated against Pydantic wrapper models — with FHIR data never persisted to the SmartHandoff database,
**So that** agents receive type-safe, validated clinical data in their working memory without the compliance risk of retaining a FHIR data copy.

---

## Business Value

Typed Pydantic models guarantee that agents receive well-formed FHIR data or a clear validation error — never a runtime `KeyError` mid-agent-task. The non-persistence constraint (AIR-012 / Constraint C-03) keeps the system outside FHIR-as-source-of-truth territory, avoiding complex synchronisation obligations.

---

## Acceptance Criteria

### Scenario 1: Patient fetched by MRN returns typed PatientModel
**Given** a patient with MRN `MRN-001` exists in the FHIR server
**When** `FHIRClient.get_patient_by_mrn("MRN-001")` is called
**Then** a `PatientModel` Pydantic object is returned with validated `id`, `name`, `birthDate`, `gender`, and `identifier` (MRN) fields.

### Scenario 2: MRN not found falls back to name + DOB search
**Given** no patient with MRN `MRN-UNKNOWN` exists in FHIR
**When** `FHIRClient.get_patient_by_mrn("MRN-UNKNOWN")` is called with name `"Smith"` and DOB `"1980-01-01"` as fallback parameters
**Then** FHIR `Patient?family=Smith&birthdate=1980-01-01` is queried; if found, a partial `PatientModel` is returned with `partial_match=True`; if not found, a `PatientNotFoundWarning` is logged and `None` is returned.

### Scenario 3: FHIR data is not persisted to SmartHandoff DB
**Given** a completed agent task that fetched FHIR MedicationStatement resources
**When** the task completes and the agent scope is cleared
**Then** no rows exist in `medication` table for the fetched data; FHIR data only existed in the agent's in-memory Pydantic models during task execution.

### Scenario 4: Invalid FHIR resource raises validation error
**Given** the FHIR server returns a `MedicationStatement` with a missing mandatory `medication` field
**When** the Pydantic model validates the response
**Then** a `FHIRValidationError` is raised with the field path and received value; the agent task is marked `FAILED` with the error detail.

- [ ] All models validated against FHIR R4 resource schemas (using `fhir.resources` library)
- [ ] Non-persistence enforced: unit tests confirm no ORM `session.add()` calls occur in FHIR fetch code paths
- [ ] MRN fallback logic with `partial_match` flag
- [ ] Unit tests with FHIR R4 JSON test fixtures for all 7 resource types
- [ ] Code reviewed and approved

---

## Technical Notes

- Use `fhir.resources` Python library for base FHIR model definitions; wrap with Pydantic for custom validation
- FHIR search parameters: `Patient?identifier={system}|{mrn}` for MRN lookup
- Response bundles: extract `Bundle.entry[].resource` for list resources (Medication, Condition, etc.)
- Rate limiter (AIR-013) implemented as a token bucket decorator on all `FHIRClient` methods (100 req/min)
