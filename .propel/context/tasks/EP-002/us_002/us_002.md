# US-002: Add Retry, Circuit Breaker, and Rate Limiting to FHIR Client

> **Epic:** EP-002 — EHR / FHIR Integration | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have

## User Story

**As a** backend engineer,
**I want** the FHIR client to automatically retry transient failures and open a circuit breaker after repeated EHR outages,
**so that** brief EHR downtime doesn't cascade into agent failures and extended outages are handled gracefully.

## Acceptance Criteria

- [ ] **AC-1:** Transient HTTP errors (5xx, timeout) retried with exponential backoff: attempt 1 after 1s, attempt 2 after 2s, attempt 3 after 4s; 4th failure raises `FHIRUnavailableError`
- [ ] **AC-2:** Circuit breaker opens after 10 consecutive failures within 60 seconds: all subsequent calls immediately raise `FHIRCircuitOpenError` without making HTTP requests; circuit half-opens after 120 seconds for a probe request
- [ ] **AC-3:** On `FHIRUnavailableError`: agent creates a partial encounter record (demographics from HL7 PID only) and logs `WARNING` with encounter_id; care coordinator notified via SignalR
- [ ] **AC-4:** Circuit breaker state visible in health endpoint `GET /ready` response: `{"fhir_circuit": "OPEN|CLOSED|HALF_OPEN"}`
- [ ] **AC-5:** Unit tests verify retry sequence and circuit breaker state transitions using `httpx.MockTransport`

## Definition of Done

- [ ] Retry and circuit breaker verified with integration tests using a controlled failing mock server
- [ ] Circuit breaker state exposed in `GET /ready` response and visible in Cloud Monitoring dashboard
