# US-004: Patient Resolution by MRN with Name/DOB Fallback

> **Epic:** EP-002 — EHR / FHIR Integration | **Sprint:** 1 | **Points:** 2 | **Priority:** Must Have

## User Story

**As a** backend engineer,
**I want** the FHIR patient resolver to first look up by MRN and fall back to name + date-of-birth if MRN is absent or unresolved,
**so that** the maximum number of patients can be matched to FHIR records even in imperfect data scenarios.

## Acceptance Criteria

- [ ] **AC-1:** Primary resolution: `GET {fhir_base}/Patient?identifier={system}|{mrn}` — returns match if 1 result; raises `FHIRAmbiguousPatientError` if >1 result; falls back if 0 results
- [ ] **AC-2:** Fallback resolution: `GET {fhir_base}/Patient?family={last_name}&given={first_name}&birthdate={dob}` — returns match if 1 result; raises `FHIRAmbiguousPatientError` if >1 result
- [ ] **AC-3:** If both attempts return 0 results: logs `WARNING` with encounter_id and HL7 source MRN; creates partial encounter record from HL7 PID data; fires care coordinator alert via SignalR
- [ ] **AC-4:** `FHIRAmbiguousPatientError` logs the number of matching patients (not their PHI) and notifies admissions clerk via SignalR; encounter processing continues with partial data
- [ ] **AC-5:** Unit tests cover: exact MRN match, MRN miss → name/DOB match, MRN miss → name/DOB miss, ambiguous results for both paths

## Definition of Done

- [ ] All 5 test scenarios pass with mock FHIR responses
- [ ] Partial encounter creation verified: encounter exists in DB with `status=ADMITTED` even when FHIR lookup fails

## Dependencies

- EP-002 US-001 (FHIR client with authentication)
- EP-DATA US-001 (Encounter ORM model for partial record creation)
