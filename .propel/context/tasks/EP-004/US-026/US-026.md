# US-026: Validate Discharge Document Completeness Before Review

> **Epic:** EP-004 — Documentation Agent | **Sprint:** 2 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** physician reviewing a discharge summary,
**I want to** be notified before I review a document if any required clinical sections are missing or empty,
**So that** I don't approve an incomplete document that could leave a patient without critical discharge instructions.

---

## Business Value

Clinical documentation completeness is a regulatory requirement (Joint Commission) and a patient safety imperative. A discharge summary missing `warning_signs` or `follow_up_instructions` could cause a patient to miss a red flag symptom. Automated completeness validation is the safety net before physician review.

---

## Acceptance Criteria

### Scenario 1: Complete document passes validation and status set to PENDING_REVIEW
**Given** a generated discharge summary with all required fields populated
**When** the completeness validator runs
**Then** the `Document.completeness_status=COMPLETE` and `Document.status=PENDING_REVIEW`; the document appears in the physician review queue.

### Scenario 2: Document with missing mandatory section flagged as INCOMPLETE
**Given** a generated summary is missing the `follow_up_instructions` section
**When** the completeness validator runs
**Then** `Document.completeness_status=INCOMPLETE`, `Document.missing_fields=["follow_up_instructions"]`; the document is held in `DRAFT` status and NOT shown in the physician review queue until resolved.

### Scenario 3: Required fields checklist is configurable
**Given** the required fields list is defined in application config
**When** a new required field `"specialist_referral"` is added to the config
**Then** the validator immediately checks for that field in all future documents without code changes.

### Scenario 4: Completeness check result is included in task status API
**Given** the completeness validator has run
**When** `GET /api/v1/encounters/{id}/tasks` is called
**Then** the `DOCUMENTATION` task status includes `completeness_status` and `missing_fields` array.

---

## Definition of Done

- [ ] `CompletenessValidator` class with configurable required fields list (YAML config)
- [ ] Required fields for discharge summary: `diagnosis_summary`, `medications_at_discharge`, `follow_up_instructions`, `warning_signs`, `activity_restrictions`
- [ ] `Document.completeness_status` field: `COMPLETE` or `INCOMPLETE`
- [ ] `Document.missing_fields` JSONB field listing absent sections
- [ ] Documents with `INCOMPLETE` status NOT returned in review queue API
- [ ] Unit tests: complete doc, missing one field, missing multiple fields
- [ ] Code reviewed and approved

---

## Technical Notes

- Validator runs as a post-generation step within the `DocumentationAgent.process()` method
- Empty string and `null` values for required fields both count as missing
- YAML config file: `config/document_completeness.yaml` with `required_fields:` list per document type
