# US-025: Generate AI Discharge Summary from FHIR Data within 30 Seconds

> **Epic:** EP-004 — Documentation Agent | **Sprint:** 2 | **Points:** 8 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** physician (Dr. David),
**I want to** receive an AI-generated discharge summary within 30 seconds of a discharge event (A03), populated from the patient's FHIR encounter data,
**So that** I spend <30 minutes finalising the discharge rather than 2–4 hours writing it from scratch.

---

## Business Value

Discharge documentation is the #1 time sink for physicians — averaging 2–4 hours per patient. AI-generated summaries directly deliver BO-03 (60% documentation time reduction). The 30-second SLA ensures the summary is ready by the time the physician opens the review screen, eliminating wait time.

---

## Acceptance Criteria

### Scenario 1: Discharge summary generated within 30 seconds for 95% of cases
**Given** an A03 discharge event triggers the documentation agent
**When** Gemini 1.5 Pro generates the summary using FHIR encounter data
**Then** the `Document` record is created with `status=PENDING_REVIEW` within 30 seconds for 95% of test cases (p95 latency threshold).

### Scenario 2: Template fallback triggered at 28-second timeout
**Given** the Gemini API call takes more than 28 seconds
**When** the 28-second timeout fires
**Then** a template-based discharge summary is generated from structured FHIR fields; the document is created with `generation_type=TEMPLATE` and `status=PENDING_REVIEW`; no exception is thrown.

### Scenario 3: Generated summary contains required clinical sections
**Given** a patient with diabetes, hypertension, and prescribed metformin and lisinopril
**When** the documentation agent generates the discharge summary
**Then** the structured output includes all mandatory sections: `diagnosis_summary`, `procedures`, `medications_at_discharge`, `follow_up_instructions`, `warning_signs`, `activity_restrictions`.

### Scenario 4: Minimum-necessary PHI in LLM prompt
**Given** the Jinja2 prompt template is rendered for the LLM call
**When** the prompt content is logged at DEBUG level (audit log only)
**Then** the prompt contains ICD-10 codes and generic descriptions — NOT the patient's full name, address, phone number, or SSN; only encounter ID and clinically necessary identifiers are included.

---

## Definition of Done

- [ ] `DocumentationAgent` extends `BaseAgent`; processes A03 (discharge) and A02 (transfer) ADT events
- [ ] FHIR data fetcher: calls `FHIRClient` for `Patient`, `Encounter`, `Condition`, `MedicationStatement` resources
- [ ] Jinja2 prompt template: `prompts/discharge_summary.jinja2` with PHI-minimised context
- [ ] Vertex AI Gemini 1.5 Pro call with `response_schema` = `DischargeSummarySchema` Pydantic model
- [ ] 25-second API timeout; template fallback at 28 seconds (2-second buffer for template generation)
- [ ] `Document` ORM record created: `status=PENDING_REVIEW`, `ai_assisted_label=True`, `content=encrypted(summary_json)`
- [ ] Performance test: p95 generation latency <30 seconds across 100 test cases
- [ ] PHI audit test: confirm prompt contains no PII fields beyond minimum-necessary set
- [ ] Code reviewed and approved

---

## Technical Notes

- Vertex AI Gemini 1.5 Pro: use `generation_config.response_mime_type="application/json"` for structured output
- LangChain chain: `prompt_template | gemini_model.with_structured_output(DischargeSummarySchema)`
- Template fallback: Jinja2 template rendering from FHIR fields — no LLM call, deterministic output
- Streaming: use `generate_content_async` with streaming for incremental token delivery (reduces perceived latency)
