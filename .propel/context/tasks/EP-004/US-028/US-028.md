# US-028: Provide Dual-Pane Document Review UI with Change Tracking

> **Epic:** EP-004 — Documentation Agent | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** physician (Dr. David) reviewing an AI-generated discharge summary,
**I want to** see the original AI draft on the left and an editable version on the right with tracked changes highlighted,
**So that** I can efficiently compare, edit, and finalise the document with a clear record of what was changed from the AI original.

---

## Business Value

Change tracking creates an auditable trail of AI-to-clinician modifications, satisfying HIPAA documentation requirements and malpractice defensibility. The dual-pane UX reduces review time by eliminating the need to switch between views, directly contributing to the 60% documentation time reduction target (BO-03).

---

## Acceptance Criteria

### Scenario 1: AI draft displayed on left, editable pane on right
**Given** a physician opens the document review page for encounter `ENC-001`
**When** the page loads
**Then** the left pane shows the immutable AI-generated draft; the right pane shows an editable version pre-populated with the same content; both panes scroll in sync.

### Scenario 2: Every clinician edit is tracked with author and timestamp
**Given** Dr. David edits the `medications_at_discharge` section in the right pane
**When** the change is auto-saved (debounced 2 seconds)
**Then** a JSON diff entry is appended to `Document.change_log`: `{"field": "medications_at_discharge", "old_value": "...", "new_value": "...", "author_id": "...", "timestamp": "..."}`.

### Scenario 3: Document save persists changes without approval
**Given** Dr. David has made edits but not yet clicked "Approve"
**When** the "Save Draft" button is clicked
**Then** the edited content is persisted to `Document.content` (encrypted); `status` remains `PENDING_REVIEW`; the AI-Assisted label remains visible.

### Scenario 4: Approve action requires physician role
**Given** a nurse attempts to call `PATCH /api/v1/documents/{id}/approve`
**When** the API request is made with a nurse-role JWT
**Then** the response is `403 Forbidden`; the document status remains `PENDING_REVIEW`.

---

## Definition of Done

- [ ] Angular `DocumentReviewComponent` with dual-pane layout (left: read-only, right: editable)
- [ ] Angular `QuillEditorModule` or `ng-prosemirror` for the editable pane with structured section editing
- [ ] Auto-save: 2-second debounce on content change → `PATCH /api/v1/documents/{id}` with diff payload
- [ ] Change log displayed below the editable pane: timeline of author, timestamp, field changed
- [ ] "Approve" button visible only to physician role; "Reject" button to all reviewers
- [ ] `PATCH /api/v1/documents/{id}/approve` API endpoint with RBAC check (physician only)
- [ ] `PATCH /api/v1/documents/{id}/reject` API endpoint with `rejection_reason` body field
- [ ] Accessibility: dual pane keyboard navigable; WCAG 2.1 AA compliant (axe-core verified)
- [ ] Code reviewed and approved

---

## Technical Notes

- Left pane must be read-only at all times — use `contenteditable=false` or a non-editable viewer component
- JSON diff computation: `jsondiff` library or custom field comparison for structured sections
- Scroll sync: listen to `scroll` event on one pane and mirror `scrollTop` on the other (debounced)
- Change log stored as a JSONB array in `Document.metadata.change_log` column
