# US-052: Implement OTP Passwordless Authentication for Patient Portal

> **Epic:** EP-010 — Patient Portal | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** discharged patient (Pat),
**I want to** access my discharge instructions by clicking the link in my SMS, entering a one-time code, and receiving access within 30 seconds — without needing to create an account or remember a password,
**So that** I can easily access my care information from my phone even if I am elderly, have low tech literacy, or am feeling unwell.

---

## Business Value

OTP authentication reduces portal access friction to the minimum possible — a single SMS link and 6-digit code. Password-based portals see 40–60% abandonment from patients who forget their credentials. Low-friction access directly increases portal utilisation, which reduces post-discharge phone calls to the hospital (BO-04).

---

## Acceptance Criteria

### Scenario 1: OTP authentication flow completes within 30 seconds
**Given** a patient taps the SMS portal link and enters their OTP
**When** `POST /api/v1/auth/patient/verify` is called with the correct OTP
**Then** a patient-scoped JWT (60-minute expiry, `encounter_id` claim) is returned within 30 seconds of the SMS link being tapped.

### Scenario 2: Rate limit blocks 6th OTP request within 1 hour
**Given** a patient has requested 5 OTPs within 60 minutes
**When** a 6th OTP request is made via `POST /api/v1/auth/patient/otp`
**Then** the response is `429 Too Many Requests` with `Retry-After` header; no OTP is generated or sent.

### Scenario 3: OTP expires after 10 minutes
**Given** an OTP was generated 11 minutes ago
**When** the patient submits the OTP via `POST /api/v1/auth/patient/verify`
**Then** the response is `401 Unauthorized` with message "OTP has expired. Please request a new code."; no JWT is issued.

### Scenario 4: Patient JWT is scoped to own encounter only
**Given** a patient JWT has `encounter_id=ENC-001` claim
**When** the patient calls any API with `encounter_id=ENC-002` in the request
**Then** the response is `403 Forbidden`; the middleware validates JWT `encounter_id` claim matches the requested resource.

---

## Definition of Done

- [ ] `POST /api/v1/auth/patient/otp` — generates 6-digit OTP, hashes with bcrypt, stores in Redis (`otp:{portal_token}` key, TTL=600s), triggers Twilio Verify send
- [ ] `POST /api/v1/auth/patient/verify` — validates OTP hash from Redis, issues patient JWT on success
- [ ] OTP rate limit: Redis counter `otp_attempts:{portal_token}` with TTL=3600s (1 hour); block at 5 attempts
- [ ] Patient JWT: signed with HS256, claims: `sub=patient_id`, `encounter_id`, `exp=60min`
- [ ] Angular OTP input component: `PatientOtpComponent` with 6-digit auto-advance input, countdown timer
- [ ] Unit tests: OTP expiry, rate limit, scope enforcement
- [ ] Code reviewed and approved

---

## Technical Notes

- OTP: store as `bcrypt(otp, salt)` in Redis — NOT plaintext; verify via `bcrypt.checkpw()`
- Portal token: signed JWT in the SMS link (not a UUID); encodes `patient_id` + `encounter_id` + 24h expiry
- OTP 6-digit: `secrets.randbelow(1000000)` zero-padded to 6 digits
- Angular OTP input: 6 single-character `<input>` elements with auto-focus-next on input
