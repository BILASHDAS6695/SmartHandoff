# US-001: Patient OTP Authentication for Portal Access

> **Epic:** EP-010 — Patient Portal | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have

## User Story

**As a** discharged patient,
**I want** to access my portal via an SMS link and one-time passcode — no password needed,
**so that** logging in is effortless and secure even if I've never used the system before.

## Acceptance Criteria

- [ ] **AC-1:** On discharge completion (encounter status → DISCHARGED): system generates a signed portal token (JWT, 24h expiry, encounter-scoped, signed with Secret Manager key) and sends SMS with portal URL `https://portal.smarthandoff.health?token={portal_token}`
- [ ] **AC-2:** Patient opens URL → Angular portal validates portal token signature and expiry → patient taps "Send me a passcode" → `POST /api/v1/auth/patient/otp {portal_token}` → Twilio Verify sends 6-digit SMS OTP
- [ ] **AC-3:** Patient enters OTP → `POST /api/v1/auth/patient/verify {otp, portal_token}` → OTP hash-compared in Redis → on match: patient JWT issued (60-min expiry, data scoped to own encounter_id only)
- [ ] **AC-4:** Rate limiting: 5 OTP requests per phone number per hour; 6th request returns 429 with `Retry-After: 3600` header
- [ ] **AC-5:** OTP expires after 10 minutes; expired OTP returns 401 with "Your passcode has expired — request a new one" message
- [ ] **AC-6:** OTP stored as SHA-256 hash in Redis (not plaintext); hash deleted on successful verification or expiry
- [ ] **AC-7:** Full flow from portal link click to authenticated portal in <30 seconds confirmed in UAT

## Definition of Done

- [ ] OTP flow tested end-to-end in staging with real Twilio Verify delivery
- [ ] Rate limiting verified: 6th OTP request returns 429
- [ ] SHA-256 hash confirmed in Redis (not plaintext OTP)

## Dependencies

- EP-013 US-002 (Twilio OTP delivery)
- EP-TECH US-001 (Redis for OTP hash storage)
- EP-004 US-001 (Discharge completion triggers portal link generation)
