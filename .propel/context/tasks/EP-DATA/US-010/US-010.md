# US-010: Automate Data Retention via pg_cron Jobs

> **Epic:** EP-DATA — Data Foundation & Schema | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Compliance Officer,
**I want to** automate data retention enforcement via scheduled pg_cron jobs that archive encounter data after 7 years and purge audit logs older than 6 years,
**So that** the system meets HIPAA retention requirements (BR-022, BR-023) without manual intervention and storage costs are controlled over time.

---

## Business Value

HIPAA mandates a minimum 6-year retention for medical records and audit logs. Manual purging is error-prone and audit trail gaps create compliance liability. Automated pg_cron jobs ensure continuous compliance while preventing unbounded database growth.

---

## Acceptance Criteria

### Scenario 1: Encounter archival job runs on schedule
**Given** the pg_cron job `archive-old-encounters` is registered
**When** the nightly cron fires (03:00 UTC)
**Then** all encounters with `discharge_date < NOW() - INTERVAL '7 years'` are moved to the `encounter_archive` table and the original rows are deleted; the job completion is logged to Cloud Logging.

### Scenario 2: Audit log purge job respects 6-year retention
**Given** the pg_cron job `purge-old-audit-logs` is registered
**When** the weekly cron fires
**Then** all `audit_log` rows with `created_at < NOW() - INTERVAL '6 years'` are exported to Cloud Storage WORM bucket and then deleted; rows within the retention window are untouched.

### Scenario 3: pg_cron job schedules are visible
**Given** the Alembic migration has registered the cron jobs
**When** `SELECT * FROM cron.job` is executed on the database
**Then** at least 3 jobs are listed: encounter archival, audit log purge, and materialised view refresh, each with their expected schedule expressions.

### Scenario 4: Archival failure is alerted
**Given** a pg_cron archival job fails due to a transient error
**When** the job raises an exception
**Then** an error entry appears in `cron.job_run_details` and a Cloud Monitoring alert fires within 5 minutes to notify the on-call engineer.

---

## Definition of Done

- [x] Alembic migration enables `pg_cron` extension and registers all retention jobs
- [x] `encounter_archive` table created with identical schema to `encounter` plus `archived_at` timestamp
- [x] pg_cron job for encounter archival: nightly at 03:00 UTC, moves rows older than 7 years
- [x] pg_cron job for audit log purge: weekly, exports to Cloud Storage then deletes rows older than 6 years
- [x] Cloud Storage export uses WORM (retention policy locked) bucket for audit log archives
- [x] Cloud Monitoring alert configured on `cron.job_run_details` error status
- [x] Unit tests verify archival logic with synthetic past-dated records
- [x] Code reviewed and approved

---

## Technical Notes

- pg_cron runs inside the PostgreSQL instance; Cloud SQL supports it when `cloudsql.enable_pgcron=on`
- Audit log export to GCS: use `COPY TO` with a GCS extension or a Cloud Function triggered by cron completion event
- Archive bucket must have `defaultObjectAcl: private` and a 6-year Object Retention Lock to prevent premature deletion
- Job schedule format: `'0 3 * * *'` for daily 3am UTC
