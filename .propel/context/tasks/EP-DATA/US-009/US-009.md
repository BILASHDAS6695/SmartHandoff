# US-009: Configure PgBouncer Connection Pool & Read Replica Routing

> **Epic:** EP-DATA — Data Foundation & Schema | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Backend Engineer,
**I want to** configure PgBouncer in transaction-mode (max 500 client connections) and implement SQLAlchemy session routing that directs reads to the Cloud SQL replica and writes to the primary,
**So that** the system can handle 500 concurrent users without exhausting Cloud SQL connection limits, and read-heavy analytics workloads do not impact transactional performance.

---

## Business Value

Cloud SQL PostgreSQL has a hard connection limit. Without PgBouncer, 500 concurrent users would each hold a persistent connection, crashing the database. Read replica routing offloads bed board, analytics, and dashboard queries from the write primary, protecting ADT ingestion throughput (5,000 events/day).

---

## Acceptance Criteria

### Scenario 1: PgBouncer enforces max 500 client connections
**Given** PgBouncer is running in transaction-pool mode
**When** 600 simultaneous client connections are attempted
**Then** the first 500 succeed; connections 501–600 are queued or rejected with a PgBouncer connection limit error; the Cloud SQL instance shows ≤50 server-side connections (pool_size).

### Scenario 2: Write operations route to primary
**Given** the SQLAlchemy session router is configured
**When** an `INSERT INTO encounter` ORM call is made
**Then** the query is executed on the Cloud SQL primary instance (verified via Cloud SQL audit logs showing primary instance host).

### Scenario 3: Read operations route to replica
**Given** the session router is configured
**When** a `SELECT * FROM encounter WHERE id=<id>` ORM read is executed outside a write transaction
**Then** the query is executed on the Cloud SQL read replica (verified via Cloud SQL audit logs).

### Scenario 4: Materialised views refresh within SLA
**Given** `mv_bed_board` and `mv_risk_dashboard` are created in the migration
**When** a bed status changes and the refresh trigger fires
**Then** `mv_bed_board` reflects the new status within 60 seconds; `mv_risk_dashboard` within 5 minutes.

---

## Definition of Done

- [ ] PgBouncer deployed as a sidecar container alongside Cloud Run services (or as a dedicated Cloud Run service)
- [ ] PgBouncer config: `pool_mode=transaction`, `max_client_conn=500`, `default_pool_size=20`
- [ ] SQLAlchemy `get_db_session()` dependency returns write session (primary) for mutations, read session (replica) for queries
- [ ] Materialised views created via Alembic migration: `mv_bed_board`, `mv_risk_dashboard`, `mv_kpi_daily`
- [ ] pg_cron refresh jobs scheduled: `mv_bed_board` every 60s, `mv_risk_dashboard` every 5m, `mv_kpi_daily` nightly
- [ ] Load test: 500 concurrent read queries complete without DB connection errors
- [ ] Code reviewed and approved

---

## Technical Notes

- PgBouncer as Cloud Run sidecar (Cloud Run multi-container support) keeps network latency minimal
- SQLAlchemy `sessionmaker` factory: `write_session_factory` bound to primary URL, `read_session_factory` bound to replica URL
- Use FastAPI dependency injection: `Depends(get_write_db)` vs `Depends(get_read_db)`
- pg_cron extension must be enabled on Cloud SQL (`cloudsql.enable_pgcron` flag)
