# US-002: Implement PHI Field-Level AES-256-GCM Encryption at ORM Layer

> **Epic:** EP-DATA — Data Foundation & Schema | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have

## User Story

**As a** security engineer,
**I want** all PHI columns encrypted at the application layer using AES-256-GCM before being written to PostgreSQL,
**so that** even direct database access by a DB administrator cannot expose patient data in plaintext.

## Acceptance Criteria

- [ ] **AC-1:** SQLAlchemy `TypeDecorator` subclass implements AES-256-GCM encrypt-on-write and decrypt-on-read for columns: `patient.first_name`, `patient.last_name`, `patient.dob`, `patient.phone`, `patient.email`, `document.content`, `chatbot_transcript.message`
- [ ] **AC-2:** `patient.mrn` uses **deterministic** AES-256-GCM (same plaintext always produces same ciphertext) to enable a unique index on the encrypted column for MRN-based lookups
- [ ] **AC-3:** Encryption key fetched from GCP Secret Manager at startup; key rotation is supported by version-tagging the key reference in Secret Manager
- [ ] **AC-4:** Direct SQL `SELECT first_name FROM patient` returns ciphertext bytes — confirmed in dev environment
- [ ] **AC-5:** ORM query `session.query(Patient).filter_by(mrn=mrn)` returns the correct patient record using the deterministic-encrypted MRN in the WHERE clause
- [ ] **AC-6:** Unit tests verify encrypt/decrypt round-trip for all PHI TypeDecorator columns

## Definition of Done

- [ ] PHI encryption verified in staging via direct psql query confirming ciphertext storage
- [ ] Key rotation tested: rotating secret version in Secret Manager does not break existing reads (re-encryption job documented)
- [ ] Security review sign-off from lead engineer

## Dependencies

- EP-DATA US-001 (ORM models defined)
- EP-TECH US-004 (Secret Manager configured)
