# US-004: Create Materialised Views for Dashboard Performance

> **Epic:** EP-DATA — Data Foundation & Schema | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have

## User Story

**As a** backend engineer,
**I want** materialised views pre-computing bed board, risk dashboard, and KPI data,
**so that** dashboard API queries respond in <500ms without scanning large encounter tables on every request.

## Acceptance Criteria

- [ ] **AC-1:** `mv_bed_board` materialised view contains: bed_id, unit, room, bed_label, bed_type, status, current_encounter_id, predicted_discharge_at — refreshes every 60 seconds via `REFRESH MATERIALIZED VIEW CONCURRENTLY`
- [ ] **AC-2:** `mv_risk_dashboard` materialised view contains: encounter_id, patient_mrn (encrypted), unit, risk_score, risk_tier, admit_date — refreshes every 5 minutes
- [ ] **AC-3:** `mv_kpi_daily` materialised view contains: date, avg_discharge_doc_time_minutes, readmission_rate_30d, med_reconciliation_completion_rate, bed_utilisation_pct — refreshed nightly via pg_cron
- [ ] **AC-4:** Dashboard API `GET /api/v1/dashboard/bed-board` queries `mv_bed_board`; response p95 <200ms under simulated load (100 concurrent requests)
- [ ] **AC-5:** PgBouncer connection pooler configured in transaction mode with `max_client_conn=500`; connection pool exhaustion test shows graceful queuing not rejection
- [ ] **AC-6:** SQLAlchemy read/write session router: all GET handler sessions routed to read replica; all write operations routed to primary

## Definition of Done

- [ ] All 3 materialised views created and refresh schedules active in dev DB
- [ ] Load test confirms p95 <200ms for bed board query under 100 concurrent users
- [ ] PgBouncer pool stats verified via `SHOW POOLS` command

## Dependencies

- EP-DATA US-001 (base tables exist)
- EP-TECH US-001 (Cloud SQL read replica provisioned)
