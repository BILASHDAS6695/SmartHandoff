# US-006: Implement PostgreSQL Schema Migrations via Alembic

> **Epic:** EP-DATA — Data Foundation & Schema | **Sprint:** 1 | **Points:** 8 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Backend Engineer,
**I want to** define and version-control the full PostgreSQL schema via Alembic migrations including all domain tables, indexes, unique constraints, and the encounter state machine,
**So that** every environment can be bootstrapped from scratch in a deterministic, repeatable way and schema drift is impossible.

---

## Business Value

A version-controlled schema is the foundation for every feature epic. Without deterministic migrations, CI/CD pipelines cannot create isolated test databases, hotfixes cannot be deployed safely, and the encounter state machine cannot enforce clinical workflow integrity (DR-023).

---

## Acceptance Criteria

### Scenario 1: Clean migration applies without errors
**Given** a fresh Cloud SQL PostgreSQL instance (empty DB)
**When** `alembic upgrade head` is executed
**Then** all tables are created with correct columns, data types, and constraints; zero errors are raised; and `alembic current` shows `head`.

### Scenario 2: Encounter state machine enforced at ORM layer
**Given** an encounter with `status=DISCHARGED`
**When** an application attempts to set `status=ADMITTED` via the ORM (not a valid A13 cancel scenario)
**Then** a `EncounterStateTransitionError` (409) is raised before any DB write occurs; the encounter status remains `DISCHARGED`.

### Scenario 3: MRN unique constraint prevents duplicate patients
**Given** patient with encrypted MRN `ENC(MRN-123)` already exists
**When** a new patient record is inserted with the same MRN
**Then** a `UniqueConstraintViolation` is raised by the DB and the duplicate record is not persisted.

### Scenario 4: Soft delete preserves patient records
**Given** a patient record exists with `deleted_at=NULL`
**When** the patient soft-delete API is called
**Then** `deleted_at` is set to the current timestamp; the record is excluded from standard queries but retrievable with an explicit `include_deleted=True` filter.

---

## Definition of Done

- [ ] Alembic migration files created for all 10 tables: `patient`, `encounter`, `adt_event`, `medication`, `agent_task`, `document`, `bed`, `app_user`, `audit_log`, `chatbot_transcript`
- [ ] SQLAlchemy ORM models defined for all tables with correct column types, relationships, and constraints
- [ ] Encounter state machine implemented as an ORM event listener validating allowed transitions
- [ ] `source_message_id` unique constraint on `adt_event` table (DR-022)
- [ ] Soft-delete columns (`deleted_at`) on `patient` and `encounter` tables (DR-005)
- [ ] `alembic downgrade -1` tested for each migration (reversible)
- [ ] Indexes on `encounter.status`, `encounter.risk_tier`, `patient.mrn_encrypted` for query performance (DR-004)
- [ ] Code reviewed and approved

---

## Technical Notes

- Use `alembic` with `autogenerate = True` disabled — all migrations hand-authored for precision
- SQLAlchemy 2.x with `declarative_base`; use `mapped_column` and `Mapped` type annotations
- Encounter allowed transitions: `ADMITTED→TRANSFERRED`, `ADMITTED→DISCHARGED`, `TRANSFERRED→DISCHARGED`, `DISCHARGED→ADMITTED` (only on A13 cancel)
- `alembic.ini` must use Secret Manager reference for DB connection string, not hardcoded value
