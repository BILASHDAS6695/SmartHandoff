# US-003: Enforce Audit Log Immutability via PostgreSQL Row Security

> **Epic:** EP-DATA — Data Foundation & Schema | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have

## User Story

**As a** compliance officer,
**I want** the `audit_log` table to be strictly append-only with no application role able to modify or delete records,
**so that** the audit trail is tamper-proof and satisfies HIPAA 6-year retention requirements.

## Acceptance Criteria

- [ ] **AC-1:** PostgreSQL row-level security (RLS) enabled on `audit_log`; policy `DENY DELETE, UPDATE` for all application roles (`api_writer`, `agent_writer`, etc.)
- [ ] **AC-2:** Only the `audit_writer` role can `INSERT`; no other operation is permitted for any application role — confirmed via `psql` test: `DELETE FROM audit_log WHERE audit_id = '...'` returns `ERROR: permission denied`
- [ ] **AC-3:** A superuser (human DBA) can still read but cannot delete within normal operations; deletion requires a break-glass procedure with documented approval
- [ ] **AC-4:** Nightly pg_cron job exports `audit_log` records older than 30 days to a Cloud Storage WORM bucket (`gsutil retention set`); exported rows kept in DB for 6 years total per BR-023
- [ ] **AC-5:** Audit log API `GET /api/v1/admin/audit` returns paginated records filtered by date, user, and entity type; PHI fields masked for non-compliance roles

## Definition of Done

- [ ] RLS policy verified in staging: application service account cannot DELETE from `audit_log`
- [ ] Cloud Storage WORM bucket policy confirmed (Object Lifecycle Management with retention lock)
- [ ] Integration test: 100 audit log entries written, 0 deletable via API

## Dependencies

- EP-DATA US-001 (audit_log table schema defined)
- EP-TECH US-001 (Cloud Storage bucket provisioned)
