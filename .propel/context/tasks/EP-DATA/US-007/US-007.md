# US-007: Implement AES-256-GCM PHI Field-Level Encryption

> **Epic:** EP-DATA ŌĆö Data Foundation & Schema | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Security Engineer,
**I want to** encrypt all PHI columns (`first_name`, `last_name`, `dob`, `phone`, `email`, `mrn`) at the SQLAlchemy ORM layer using AES-256-GCM with keys sourced from Secret Manager,
**So that** even if the Cloud SQL database is directly queried, patient identity data is unreadable without the encryption key.

---

## Business Value

AES-256 field-level encryption is a HIPAA technical safeguard (45 CFR ┬¦164.312) that protects PHI against database-level breaches. Using deterministic encryption for `mrn` enables unique-indexed deduplication without exposing the plaintext value, preventing duplicate patient records (DR-020).

---

## Acceptance Criteria

### Scenario 1: PHI fields are stored as ciphertext in the database
**Given** a patient record is created via the ORM with `first_name="John"`, `last_name="Doe"`, `mrn="MRN12345"`
**When** a direct SQL query `SELECT first_name, last_name, mrn FROM patient WHERE id=<id>` is executed
**Then** the returned values are ciphertext strings (base64-encoded AES-256-GCM output), not the plaintext values.

### Scenario 2: PHI fields decrypt transparently when accessed via ORM
**Given** a patient record with encrypted PHI exists in the database
**When** the ORM model is loaded and `patient.first_name` is accessed
**Then** the decrypted plaintext value `"John"` is returned without any explicit decryption call by the caller.

### Scenario 3: MRN deterministic encryption enables unique constraint
**Given** patient A with `mrn="MRN12345"` already exists
**When** a new patient with `mrn="MRN12345"` is inserted
**Then** a `UniqueConstraintViolation` is raised because both records produce the same deterministic ciphertext for the MRN column.

### Scenario 4: Encryption key rotation does not break existing records
**Given** a new encryption key version is deployed to Secret Manager
**When** the re-encryption background job runs
**Then** all existing PHI records are re-encrypted with the new key; decryption via ORM continues to work; zero data loss.

---

## Definition of Done

- [ ] SQLAlchemy `TypeDecorator` classes implemented for: `EncryptedString` (non-deterministic) and `DeterministicEncryptedString` (for MRN)
- [ ] PHI columns on `patient` and `encounter` tables use the appropriate TypeDecorators
- [ ] AES-256-GCM used with a unique random IV per non-deterministic encryption operation
- [ ] Deterministic encryption for `mrn` uses HMAC-SHA256-derived IV (reproducible, collision-resistant)
- [ ] Encryption key loaded from Secret Manager at application startup; key cached in memory (not logged)
- [ ] Unit tests confirm: (a) raw DB values are ciphertext, (b) ORM access returns plaintext
- [ ] Re-encryption utility script documented in `infra/BOOTSTRAP.md`
- [ ] Code reviewed and approved by Security Engineer

---

## Technical Notes

- Use `cryptography` library (`Fernet` is NOT suitable ŌĆö AES-256-GCM is required per BR-020)
- AESGCM from `cryptography.hazmat.primitives.ciphers.aead`
- Key size: 256-bit (32 bytes); nonce size: 96-bit (12 bytes) per GCM standard
- Deterministic MRN: derive nonce from `HMAC-SHA256(key, mrn_plaintext)[:12]` ŌĆö same plaintext always produces same ciphertext
- Store ciphertext as: `base64(nonce + tag + ciphertext)` in VARCHAR column (max ~256 chars for typical PHI fields)
