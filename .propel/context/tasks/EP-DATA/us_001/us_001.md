# US-001: Create Full PostgreSQL Schema via Alembic Migrations

> **Epic:** EP-DATA — Data Foundation & Schema | **Sprint:** 1 | **Points:** 5 | **Priority:** Must Have

## User Story

**As a** backend engineer,
**I want** all database tables created and managed via Alembic version-controlled migrations,
**so that** schema changes are auditable, reversible, and applied consistently across all environments.

## Acceptance Criteria

- [ ] **AC-1:** `alembic upgrade head` creates all tables from a clean database with zero errors: `patient`, `encounter`, `adt_event`, `medication`, `agent_task`, `document`, `bed`, `app_user`, `audit_log`, `chatbot_transcript`
- [ ] **AC-2:** `alembic downgrade -1` reverses each migration cleanly
- [ ] **AC-3:** All foreign key relationships, NOT NULL constraints, and unique constraints defined in migration SQL match the ERD in model.md
- [ ] **AC-4:** Composite indexes created: `(patient_id, admit_date DESC)` on encounter; `(unit, status)` on bed; `(risk_tier, status)` on encounter
- [ ] **AC-5:** Soft-delete `deleted_at` column present on `patient` and `encounter` tables; default query scope in ORM excludes soft-deleted rows
- [ ] **AC-6:** Alembic migration step runs as a pre-deploy Cloud Build step in CI/CD; deployment blocked if migration fails
- [ ] **AC-7:** SQLAlchemy ORM model classes defined for all 10 tables with correct column types, relationships, and repr methods

## Definition of Done

- [ ] Migrations run successfully against Cloud SQL dev instance
- [ ] `alembic current` shows correct head revision in all environments
- [ ] ORM models unit-tested with SQLite in-memory DB for basic CRUD operations

## Dependencies

- EP-TECH US-001 (Cloud SQL instance provisioned)
