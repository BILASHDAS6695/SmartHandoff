# US-008: Enforce Audit Log Immutability via PostgreSQL Row Security

> **Epic:** EP-DATA — Data Foundation & Schema | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Compliance Officer,
**I want to** ensure the `audit_log` table is append-only — all DELETE and UPDATE operations are denied at the database level for the application role — so that the PHI access audit trail is tamper-proof for the required 6-year retention period.

---

## Business Value

HIPAA requires an unalterable audit trail of all PHI access (45 CFR §164.312(b)). PostgreSQL row-level security enforces immutability even if the application is compromised, providing an independent tamper-proof record that satisfies Joint Commission and HIPAA audits.

---

## Acceptance Criteria

### Scenario 1: Application role cannot DELETE audit log rows
**Given** the row security policy is active on `audit_log`
**When** the `app_write` database role executes `DELETE FROM audit_log WHERE id=<id>`
**Then** PostgreSQL raises a permission denied error and zero rows are deleted.

### Scenario 2: Application role cannot UPDATE audit log rows
**Given** the row security policy is active on `audit_log`
**When** the `app_write` database role executes `UPDATE audit_log SET action='TAMPERED' WHERE id=<id>`
**Then** PostgreSQL raises a permission denied error and the row is unchanged.

### Scenario 3: INSERT into audit log succeeds
**Given** the `audit_writer` role has INSERT-only permission
**When** the audit logging middleware inserts a new `audit_log` entry for a PHI access event
**Then** the record is persisted successfully with correct `user_id`, `action`, `entity_type`, `entity_id`, `ip_address`, and `created_at` values.

### Scenario 4: Superuser can read audit log for compliance reporting
**Given** the `compliance_reader` role has SELECT-only permission on `audit_log`
**When** a compliance query selects audit entries within a date range
**Then** all rows are returned in full; no permission error is raised for the read-only role.

---

## Definition of Done

- [ ] PostgreSQL row-level security (RLS) policy on `audit_log`: DENY DELETE and UPDATE for `app_write` role
- [ ] Alembic migration creates the `audit_log` table and applies RLS policy
- [ ] Database roles created: `app_write` (INSERT on all tables except audit_log row-restrictions), `audit_writer` (INSERT-only on audit_log), `compliance_reader` (SELECT on audit_log)
- [ ] Audit log HIPAA middleware implemented: logs every PHI access (READ, WRITE, DELETE) to `audit_log` via `audit_writer` role
- [ ] Integration tests confirm DELETE/UPDATE on `audit_log` raise `InsufficientPrivilege` for `app_write` role
- [ ] pg_cron job scheduled to enforce 6-year retention by archiving rows older than 2,190 days to Cold Storage and deleting post-archive
- [ ] Code reviewed and approved

---

## Technical Notes

- RLS policy: `CREATE POLICY audit_immutable ON audit_log AS RESTRICTIVE FOR ALL TO app_write USING (false)` — effectively denies all operations except INSERT which is not governed by USING clause
- Use `GRANT INSERT ON audit_log TO audit_writer` — separate role for writes
- Middleware: FastAPI dependency that creates an `audit_log` entry after every request accessing PHI endpoints
- Do NOT log PHI field values in audit_log — only entity type, ID, action, and user identity
