# US-031: Detect Drug-Drug Interactions via RxNav with OpenFDA Fallback

> **Epic:** EP-005 — Medication Reconciliation Agent | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** clinical pharmacist,
**I want to** receive immediate alerts for HIGH-severity drug-drug interactions detected in a patient's discharge medications — checked via RxNav with OpenFDA as a fallback — within 60 seconds of reconciliation,
**So that** I can intervene before a dangerous drug combination leaves the hospital with the patient.

---

## Business Value

≥99% sensitivity for drug-drug interaction detection (FR-031) is a direct patient safety specification. Adverse drug interactions are responsible for over 100,000 hospitalisations annually in the US. Immediate pharmacist alerts for HIGH-severity interactions (BR-005) prevent medication-induced harm at the point of highest risk: discharge.

---

## Acceptance Criteria

### Scenario 1: HIGH-severity interaction triggers pharmacist alert within 60 seconds
**Given** a patient is discharged with `Warfarin 5mg` and `Aspirin 325mg` — a major interaction
**When** the drug interaction checker runs
**Then** a `PHARMACIST_ALERT` is created with `severity=HIGH`, `drug_pair=["Warfarin", "Aspirin"]`, `interaction_description` from RxNav; the alert appears in the pharmacist dashboard within 60 seconds.

### Scenario 2: Interaction result cached in Redis for 24 hours
**Given** the RxNav interaction for `Warfarin` + `Aspirin` was fetched
**When** the same drug pair is checked again within 24 hours
**Then** the result is returned from Redis cache (key: `drug-interaction:{rxcui1}:{rxcui2}`); no RxNav API call is made (confirmed by mock: RxNav not called).

### Scenario 3: OpenFDA is used when RxNav is unavailable
**Given** RxNav returns HTTP 503
**When** the interaction checker falls back to OpenFDA
**Then** OpenFDA is queried with the drug names; interaction results are returned; the source is recorded as `OPENFDA` in the alert metadata.

### Scenario 4: Graceful degradation when both APIs unavailable
**Given** both RxNav and OpenFDA return errors
**When** the offline degradation handler fires
**Then** the reconciliation record is flagged `interaction_check_status=INCOMPLETE`; a MEDIUM-priority pharmacist alert is created: "Interaction check unavailable — manual review required"; no CRITICAL alert is suppressed silently.

## Definition of Done

- [ ] `DrugInteractionChecker` service: checks all active drug pairs in the discharge medication list
- [ ] RxNav Interaction API: `GET https://rxnav.nlm.nih.gov/REST/interaction/list.json?rxcuis={cuis}` for batch lookup
- [ ] Severity mapping: `major` or `contraindicated` → HIGH; `moderate` → MEDIUM; `minor` → LOW
- [ ] Redis cache: `drug-interaction:{sorted_rxcui_pair}` with TTL=86400 (24h)
- [ ] OpenFDA fallback: `GET https://api.fda.gov/drug/label.json?search=warnings+interactions:{drug_name}`
- [ ] Offline flag: `interaction_check_status` field on reconciliation record
- [ ] `POST /api/v1/encounters/{id}/alerts` endpoint for pharmacist alert creation
- [ ] Unit tests: HIGH interaction path, cache hit, OpenFDA fallback, offline degradation
- [ ] Code reviewed and approved

---

## Technical Notes

- RxNorm CUIs required for RxNav API — normalised in US-030
- Batch interaction check: include all active discharge medications in one RxNav call (up to 50 RxCUIs)
- Redis key uses sorted CUI pair: `min(cui1, cui2):max(cui1, cui2)` to avoid key duplication for reversed pair order
- HIGH interaction alert must use the `IMMEDIATE` notification priority in Pub/Sub `notification-requests`
