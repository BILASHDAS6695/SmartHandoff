# US-033: Generate Patient-Readable Medication Change Summary

> **Epic:** EP-005 — Medication Reconciliation Agent | **Sprint:** 2 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** patient (Pat) being discharged,
**I want to** receive a plain-language summary of which medications are new, changed, or stopped — written in my preferred language — so that I know exactly which pills to take, which to stop, and which are new, without needing to decode medical terminology.

---

## Business Value

Medication confusion after discharge is a leading cause of preventable readmissions. A plain-language summary (FR-034) translates pharmacist-level reconciliation into patient-actionable instructions. Combined with the chatbot (EP-008), it gives patients 24/7 support for medication questions, directly reducing medication non-adherence.

---

## Acceptance Criteria

### Scenario 1: Medication change summary generated in plain English
**Given** a reconciliation result with 2 CONTINUED, 1 NEW (Lisinopril), 1 STOPPED (Metoprolol), and 1 DOSE_CHANGED (Metformin 500mg → 1000mg)
**When** the patient summary generator runs
**Then** the output includes: a "New Medications" section listing Lisinopril with purpose, dosing instructions, and common side effects; a "Stopped Medications" section noting Metoprolol was stopped; a "Changed Medications" section noting the Metformin dose increase.

### Scenario 2: Medical terms replaced with plain language
**Given** a discharge medication `Furosemide 40mg` (a diuretic)
**When** the patient summary is generated
**Then** the summary uses the drug's brand name and a plain description: "Furosemide (Lasix) — a water pill to reduce fluid buildup" rather than the generic clinical description.

### Scenario 3: Summary stored on Document record and linked to discharge instructions
**Given** the medication change summary has been generated
**When** the documentation agent finalises the discharge document
**Then** the medication summary is included as the `medications_section` of the patient discharge instructions document.

### Scenario 4: Summary generated in patient's preferred language
**Given** a patient's `preferred_language=es` (Spanish)
**When** the medication summary is generated
**Then** the Spanish translation is created using the same Gemini translation pipeline from EP-004 (US-027); the summary is stored under `Document.translations.es`.

---

## Definition of Done

- [ ] `MedicationSummaryGenerator` class: takes reconciliation result → Gemini prompt → plain-language JSON
- [ ] Gemini Flash prompt: "Write a patient-friendly medication summary. Use simple words. For each change, explain what the drug is for and what changed."
- [ ] Drug brand name lookup: RxNav `getDisplayTerms` API
- [ ] Output schema: `{"new": [...], "stopped": [...], "changed": [...], "continued": [...]}`
- [ ] Summary stored in `document` table as part of patient instructions section
- [ ] Language translation: reuse EP-004 translation pipeline (US-027)
- [ ] Unit tests: all reconciliation categories, brand name enrichment
- [ ] Code reviewed and approved

---

## Technical Notes

- Use Gemini Flash (not Pro) for this task — plain-language rewriting is lower complexity than clinical summary generation
- Brand name lookup cache: store in Redis with TTL=7 days (drug brand names rarely change)
- Dosing instruction format: "Take 1 tablet (500mg) twice daily with food"
