# US-011: Build MLLP TCP Listener for HL7 ADT Event Ingestion

> **Epic:** EP-001 — HL7 ADT Event Ingestion | **Sprint:** 1 | **Points:** 8 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Backend Engineer,
**I want to** implement an asyncio-based MLLP TCP listener on port 2575 that accepts up to 50 concurrent connections, returns ACK within 200ms, and returns NACK on parse failure,
**So that** the EHR system can reliably transmit HL7 ADT messages and receive timely acknowledgements that confirm safe receipt.

---

## Business Value

The MLLP listener is the single entry point for all ADT events. Its 200ms ACK SLA (AIR-001) prevents the EHR from timing out and retransmitting, which would create duplicate processing. Supporting 50 concurrent connections handles peak hospital admissions without dropping events.

---

## Acceptance Criteria

### Scenario 1: MLLP ACK returned within 200ms for valid message
**Given** the MLLP listener is running on port 2575
**When** a valid HL7 ADT^A01 message is sent over TCP
**Then** an ACK (AA) response is returned within 200ms, measured from TCP payload receipt to TCP ACK response sent.

### Scenario 2: MLLP NACK returned for unparseable message
**Given** a malformed HL7 message with a missing MSH segment
**When** the message is sent over TCP
**Then** a NACK (AE) response is returned with error code `207` ("Application error") and a structured error log entry is written to Cloud Logging.

### Scenario 3: 50 concurrent connections sustained without message loss
**Given** the MLLP listener is running
**When** a load test sends 50 simultaneous TCP connections each transmitting 100 ADT messages
**Then** all 5,000 messages are processed and ACKed; zero connections are dropped or time out.

### Scenario 4: Keep-alive connections are maintained
**Given** an EHR system holds a persistent MLLP connection
**When** no messages are sent for 30 seconds
**Then** the TCP connection remains open (keep-alive probe succeeds) and the next message is processed without reconnect overhead.

## Definition of Done

- [x] MLLP listener implemented using Python `asyncio` with `asyncio.start_server` (`app/mllp/server.py`)
- [x] MLLP framing: VT (0x0B) start block + FS (0x1C) end block + CR parsing implemented (`app/mllp/framing.py`)
- [x] `MllpFramingError(ValueError)` raised on missing VT, missing FS+CR, empty payload
- [x] ACK (MSH + MSA segment with `AA`) generated and returned within 200ms (`app/mllp/ack_builder.py`)
- [x] NACK (MSH + MSA segment with `AE` + ERR segment, code 207) generated on parse failure
- [x] ACK MSH mirrors original Sending App/Facility; MSA-2 echoes original MSH-10
- [x] `asyncio.wait_for(0.18s)` enforces 200ms ACK SLA; hl7apy runs in thread executor
- [x] Idle connection timeout 300s (AIR-004)
- [x] 1 MB message size safety cap with NACK response
- [x] Connection pool: maximum 50 concurrent MLLP connections enforced via semaphore (`backlog=100`)
- [x] TCP keep-alive enabled on all accepted connections (`SO_KEEPALIVE`, `TCP_KEEPIDLE=30`)
- [x] Prometheus metrics exposed: `hl7_messages_total`, `hl7_ack_latency_ms`, `hl7_active_connections` (`app/metrics.py`)
- [x] `GET /health`, `GET /ready`, and `GET /metrics` endpoints implemented as asyncio HTTP server (`app/health.py`)
- [x] `app/main.py` entry point — `asyncio.gather` runs MLLP + health servers concurrently
- [x] `Dockerfile` using `python:3.12-slim`, non-root user, `CMD ["python", "-m", "app.main"]`
- [x] Unit tests: `tests/unit/mllp/test_framing.py` (framing module, ≥10 cases)
- [x] Unit tests: `tests/unit/mllp/test_ack_builder.py` (ACK/NACK builder, ≥14 cases)
- [x] `pytest-asyncio` and `pytest-cov` added to `requirements.txt`

- [ ] Code reviewed and approved

## Technical Notes

- Use `hl7apy` Python library for HL7 v2.x message construction and validation
- Cloud Run MLLP service must use `--port=2575` and TCP (not HTTP) protocol — requires Cloud Run TCP support
- If Cloud Run TCP is unavailable, deploy MLLP listener on GKE with a TCP LoadBalancer service
- ACK MSH-9 = `ACK`, MSA-1 = `AA`, MSA-2 = original MSH-10 (message control ID)
