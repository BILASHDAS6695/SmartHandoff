# US-014: Publish ADT Events to Pub/Sub with Encounter Ordering

> **Epic:** EP-001 — HL7 ADT Event Ingestion | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Backend Engineer,
**I want to** publish parsed `ADTEvent` domain objects to the GCP Pub/Sub `adt-events` topic with `encounter_id` as the ordering key and JSON-serialised payload,
**So that** downstream agent subscribers receive events in correct encounter-chronological order and no event is lost between MLLP receipt and agent processing.

---

## Business Value

Pub/Sub with ordering keys ensures that an A03 (discharge) cannot be processed before the A01 (admit) for the same encounter, which would corrupt the encounter state machine. Guaranteed delivery prevents silent event drops that would leave agents unaware of patient movements.

---

## Acceptance Criteria

### Scenario 1: ADTEvent is published to Pub/Sub after DB persist
**Given** a valid ADTEvent has been parsed and persisted to `adt_event` table
**When** the Pub/Sub publisher dispatches the event
**Then** the message appears in the `adt-events` Pub/Sub topic within 1 second, with `encounter_id` as the ordering key and the full `ADTEvent` JSON as the message body.

### Scenario 2: Ordering key ensures per-encounter FIFO delivery
**Given** events A01, A02, A03 arrive in sequence for encounter `ENC-001`
**When** they are published with ordering key `ENC-001`
**Then** a subscriber pulling messages from the subscription receives them in A01 → A02 → A03 order for that encounter.

### Scenario 3: Pub/Sub publish failure triggers retry
**Given** a transient Pub/Sub API error occurs
**When** the publisher attempts to publish an ADTEvent
**Then** the publisher retries up to 3 times with exponential backoff; if all retries fail, the event is written to a local retry queue and an alert fires; the original Pub/Sub ACK to the processing pipeline is withheld until publish succeeds.

### Scenario 4: Published message contains required metadata attributes
**Given** an ADTEvent is published
**When** the Pub/Sub message attributes are inspected
**Then** attributes include: `event_type` (e.g. `ADMIT`), `encounter_id`, `patient_mrn_hash` (SHA-256 of MRN — not plaintext PHI), `timestamp`.

---

## Definition of Done

- [ ] `ADTEventPublisher` class wraps `google.cloud.pubsub_v1.PublisherClient` with ordering key support
- [ ] `ADTEvent` Pydantic model serialises to JSON for Pub/Sub message body
- [ ] Pub/Sub message attributes: `event_type`, `encounter_id`, `patient_mrn_hash`, `iso_timestamp`
- [ ] Ordering key = `encounter_id` (string) set on every `PublishRequest`
- [ ] Publisher retry: 3 attempts, exponential backoff (1s, 2s, 4s); Prometheus counter `pubsub_publish_failures_total`
- [ ] Unit tests: verify message body schema, ordering key, attribute presence
- [ ] Integration test: publish and pull from test subscription confirms FIFO order for same encounter
- [ ] Code reviewed and approved

---

## Technical Notes

- Cloud Pub/Sub ordering keys require `enable_message_ordering=True` on the subscription; set this in Terraform
- PHI must not appear in Pub/Sub message attributes (only in the encrypted body). Use `SHA-256(mrn)` for any attribute that references a patient
- Publisher uses `google-cloud-pubsub` with `PublisherOptions(enable_message_ordering=True)`
- Ordering key max length is 1,024 bytes; `encounter_id` (UUID) is well within this limit
