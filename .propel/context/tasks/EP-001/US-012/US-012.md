## User Story

**As a** Backend Engineer,
**I want to** parse HL7 v2.x ADT messages (MSH, EVN, PID, PV1, PV2, DG1 segments) into a typed `ADTEvent` domain model and route all 8 event types (A01, A02, A03, A04, A08, A11, A12, A13) to their respective handlers,
**So that** downstream agents receive a structured, validated event object regardless of which EHR system sent the HL7 message.

---

## Business Value

Parsing raw HL7 into a typed domain model is the critical transformation that makes unstructured hospital data consumable by AI agents. Without this, no agent can operate. Supporting all 8 ADT event types ensures the system handles the full patient journey from admission to cancellation.

## Acceptance Criteria

### Scenario 1: A01 message parsed to ADTEvent with all required fields
**Given** an HL7 ADT^A01 test fixture message with MSH, PID, PV1 segments
**When** the HL7 parser processes the message
**Then** the resulting `ADTEvent` has: `event_type=ADMIT`, `patient_mrn` from PID-3, `encounter_id` from PV1-19, `event_time` from EVN-2, `attending_provider` from PV1-7, `admit_reason` from PV2-3.

### Scenario 2: All 8 event types are correctly classified
**Given** test fixtures for A01, A02, A03, A04, A08, A11, A12, A13
**When** each message is parsed
**Then** the `event_type` field maps correctly: A01â†’ADMIT, A02â†’TRANSFER, A03â†’DISCHARGE, A04â†’REGISTER, A08â†’UPDATE, A11â†’CANCEL_ADMIT, A12â†’CANCEL_TRANSFER, A13â†’CANCEL_DISCHARGE.

### Scenario 3: Unknown event type returns NACK
**Given** an HL7 ADT^A99 message (unsupported event type)
**When** the parser processes the message
**Then** a NACK (AE) is returned, a structured error log entry records `unknown_event_type=A99`, and no `ADTEvent` record is created.

### Scenario 4: Missing mandatory segments trigger validation failure
**Given** an HL7 message missing the PID segment (mandatory)
**When** the validator runs
**Then** validation fails with a structured error message listing the missing segment, a NACK is returned, and the raw message is still archived to Cloud Storage for manual review.

## Definition of Done

- [ ] `HL7Parser` class implemented using `hl7apy` with extraction for: MSH-3, MSH-7, MSH-9, MSH-10; EVN-2; PID-3, PID-5, PID-7, PID-11; PV1-2, PV1-3, PV1-7, PV1-18, PV1-19; PV2-3; DG1-3
- [ ] `ADTEvent` Pydantic model defined with all extracted fields, types, and validation
- [ ] Event type routing map: `{A01: admit_handler, A02: transfer_handler, ...}` with handler registration pattern
- [ ] Mandatory segment validator: raises `HL7ValidationError` if MSH, PID, or PV1 are absent
- [ ] Unknown event types log warning and return NACK; no exception propagation to MLLP layer
- [ ] Unit tests with HL7 test fixtures for all 8 event types (fixtures stored in `tests/fixtures/hl7/`)
- [ ] Code reviewed and approved

---

## Technical Notes

- `hl7apy` mode: `VALIDATION_LEVEL.STRICT` for segment presence, `VALIDATION_LEVEL.QUIET` for optional field population
- PID-3 (patient identifier list): extract first CX component with type `MR` for MRN
- PV1-18 (patient type) and PV1-2 (patient class) used together to determine admit_type
- DG1 segment may be repeated; capture all diagnoses as a list
