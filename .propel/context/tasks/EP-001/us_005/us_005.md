# US-005: Handle ADT Cancellation Events to Halt In-Progress Workflows

> **Epic:** EP-001 — HL7 ADT Event Ingestion | **Sprint:** 2 | **Points:** 3 | **Priority:** Must Have

## User Story

**As a** system,
**I want** ADT cancellation events (A11, A12, A13) to immediately halt all in-progress agent tasks for the affected encounter and revert the encounter status,
**so that** cancelled admissions, transfers, or discharges do not produce incorrect documents, bed assignments, or notifications.

## Acceptance Criteria

- [ ] **AC-1:** On receipt of A11 (Cancel Admit): all `AgentTask` records for the encounter with `status IN (PENDING, IN_PROGRESS)` updated to `CANCELLED`; `Encounter.status` reverted to `REGISTERED` (or prior state)
- [ ] **AC-2:** On receipt of A12 (Cancel Transfer): `Encounter.unit` reverted to pre-transfer unit; both source and destination `Bed.status` reverted; `AgentTask` records for transfer workflow cancelled
- [ ] **AC-3:** On receipt of A13 (Cancel Discharge): `Encounter.status` reverted to `ADMITTED`; any PENDING `Document` records with `document_type=DISCHARGE_SUMMARY` deleted or marked `CANCELLED`; `Bed` re-assigned to patient
- [ ] **AC-4:** Cancellation event published to Pub/Sub with `event_type=A11|A12|A13`; Coordinator Agent consumes and executes cancellation workflow within 5 seconds
- [ ] **AC-5:** Dashboard notifies affected unit staff via SignalR push within 1 second of cancellation processing
- [ ] **AC-6:** Integration test: send A01 → wait 2 seconds (agents start) → send A11 → verify all agent tasks cancelled and encounter status reverted

## Definition of Done

- [ ] All 3 cancellation event types covered by integration tests
- [ ] No orphaned PENDING agent tasks remain after cancellation (verified by DB query)
- [ ] SignalR cancellation notification confirmed in browser during test

## Dependencies

- EP-001 US-004 (Pub/Sub publish working)
- EP-003 US-001 (Coordinator Agent subscribes and cancels tasks)
- EP-DATA US-001 (Encounter state machine enforced at ORM)
