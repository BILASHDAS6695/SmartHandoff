# US-004: Publish ADT Events to Pub/Sub with Idempotency

> **Epic:** EP-001 — HL7 ADT Event Ingestion | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have

## User Story

**As a** backend engineer,
**I want** ADT events published to GCP Pub/Sub with ordering keys and idempotency guards,
**so that** agents receive events in encounter order and duplicate HL7 messages are safely ignored without reprocessing.

## Acceptance Criteria

- [ ] **AC-1:** ADT event published to `adt-events` Pub/Sub topic with `ordering_key = encounter_id`; ensures per-encounter event ordering for agent subscribers
- [ ] **AC-2:** `source_message_id` (MSH-10) stored as unique constraint in `adt_event.source_message_id`; duplicate message detected → ACK returned → no Pub/Sub publish → no DB insert; idempotency verified in integration test
- [ ] **AC-3:** Pub/Sub publish retried up to 3 times with exponential backoff on transient failures; if all retries fail: message moved to `adt-events-dlq` dead-letter topic and P1 alert fired
- [ ] **AC-4:** Pub/Sub message attributes include: `event_type`, `encounter_id`, `patient_mrn` (encrypted), `event_time` — enabling agent-side routing without deserialising message body
- [ ] **AC-5:** `adt_event.processing_status` updated to `PUBLISHED` after successful Pub/Sub publish; `FAILED` on DLQ

## Definition of Done

- [ ] Integration test: send same MSH-10 message ID twice → second message produces no Pub/Sub event and no DB row
- [ ] DLQ test: simulate Pub/Sub unavailable → event lands in DLQ → P1 alert fires
- [ ] Ordering test: 5 sequential events for same encounter arrive in correct order at subscriber

## Dependencies

- EP-001 US-001 (MLLP listener)
- EP-001 US-002 (HL7 parser produces ADTEvent model)
- EP-TECH US-001 (Pub/Sub topic and DLQ provisioned)
