# US-015: Handle ADT Cancellation Events (A11/A12/A13)

> **Epic:** EP-001 — HL7 ADT Event Ingestion | **Sprint:** 2 | **Points:** 5 | **Priority:** Must Have
> **Status:** Done | **Date:** 2026-07-22

---

## User Story

**As a** Backend Engineer,
**I want to** handle HL7 cancellation events (A11 — cancel admit, A12 — cancel transfer, A13 — cancel discharge) by halting in-progress agent workflows and reverting the encounter status,
**So that** erroneous admissions or transfers are corrected in the system before AI agents act on invalid patient data.

---

## Business Value

Cancellation events are patient safety features: an A11 (cancel admit) means the patient was never actually admitted and agents should not generate discharge summaries or medication reconciliation for a non-existent encounter. Handling these prevents incorrect AI-generated clinical documents from reaching care teams.

---

## Acceptance Criteria

### Scenario 1: A11 (cancel admit) halts all pending agent tasks
**Given** an A01 was received for encounter `ENC-001` and 5 `AgentTask` records are in `PENDING` or `IN_PROGRESS` status
**When** an A11 cancellation event arrives for the same encounter
**Then** all 5 `AgentTask` records are updated to `CANCELLED`, the encounter status reverts to `PRE_ADMISSION`, and a SignalR notification is broadcast to care team dashboard users.

### Scenario 2: A12 (cancel transfer) reverts encounter to previous unit
**Given** an A02 transfer event moved encounter `ENC-002` from unit `3A` to `4B`
**When** an A12 cancellation event arrives
**Then** the encounter's `current_unit` reverts to `3A`, the transfer record is marked `CANCELLED`, and any in-progress Transfer-Note agent tasks are cancelled.

### Scenario 3: A13 (cancel discharge) allows re-admit
**Given** an A03 discharge event was processed for encounter `ENC-003`
**When** an A13 cancellation event arrives
**Then** the encounter status reverts to `ADMITTED`, the discharge document is marked `CANCELLED` (not deleted), and the state machine permits the next A01 for re-admission.

### Scenario 4: Cancellation event for unknown encounter is logged and ignored
**Given** an A11 arrives for encounter `ENC-UNKNOWN` that does not exist in the database
**When** the handler processes the event
**Then** an ACK (AA) is returned, a warning log entry is created with `unknown_encounter_id=ENC-UNKNOWN`, and no state changes are made.

---

## Definition of Done

- [ ] Cancellation event handlers implemented for A11, A12, A13 in the event routing map
- [ ] `cancel_agent_tasks(encounter_id)` service method updates all non-terminal `AgentTask` records to `CANCELLED`
- [ ] Encounter state machine extended to allow: `ADMITTED→PRE_ADMISSION` (A11), `TRANSFERRED→ADMITTED` (A12), `DISCHARGED→ADMITTED` (A13)
- [ ] SignalR notification published on cancellation (payload: `{event: ENCOUNTER_CANCELLED, encounter_id, reason}`)
- [ ] Cancelled documents retain their content with `status=CANCELLED` — no hard delete
- [ ] Unit tests: each cancellation type with correct state reversion
- [ ] Integration test: A01 → 5 tasks created → A11 → all tasks CANCELLED
- [ ] Code reviewed and approved

---

## Technical Notes

- Use database transaction to ensure encounter status update + task cancellations are atomic
- SignalR notification dispatched via async background task after transaction commits (do not block MLLP ACK)
- A11 should also publish a `WORKFLOW_CANCELLED` event to Pub/Sub so the agent framework (EP-003) can drain its own in-flight tasks
- If an agent is mid-LLM call when A11 arrives, the agent must check a cancellation flag before persisting its output
