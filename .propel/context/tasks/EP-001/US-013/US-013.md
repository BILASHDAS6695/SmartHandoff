# US-013: Archive Raw HL7 Messages & Enforce Idempotency

> **Epic:** EP-001 — HL7 ADT Event Ingestion | **Sprint:** 1 | **Points:** 3 | **Priority:** Must Have
> **Status:** Draft | **Date:** 2026-07-14

---

## User Story

**As a** Backend Engineer,
**I want to** archive every raw HL7 message to a HIPAA-compliant Cloud Storage bucket before sending the ACK, and deduplicate messages using the MSH-10 message control ID as an idempotency key,
**So that** no HL7 message is ever lost even if downstream processing fails, and retransmitted messages from the EHR are safely ignored rather than double-processed.

---

## Business Value

Archiving before ACK (AIR-003) guarantees that raw messages exist for forensic replay even if the database write fails. Idempotency (DR-022) prevents a patient being admitted twice in the system due to EHR retry behaviour, which would trigger duplicate agent workflows and confuse clinical staff.

---

## Acceptance Criteria

### Scenario 1: Raw HL7 message archived before ACK is returned
**Given** a valid HL7 ADT^A01 message arrives
**When** the MLLP handler processes it
**Then** the raw HL7 text is written to `gs://hl7-archive/{YYYY-MM-DD}/{msg-control-id}.hl7` before the ACK TCP response is sent; the Cloud Storage write is confirmed (GCS object exists).

### Scenario 2: Duplicate MSH-10 message ID is detected and skipped
**Given** an HL7 message with MSH-10 `MSG-20260714-001` has already been processed and stored in `adt_event.source_message_id`
**When** the same MSH-10 value arrives again
**Then** an ACK (AA) is returned immediately (the message is considered already-received), no new `adt_event` record is created, and a `duplicate_message_skipped` log entry is written.

### Scenario 3: Archive path is date-partitioned
**Given** messages arrive on different calendar dates
**When** the GCS archive objects are listed
**Then** objects are stored under date-partitioned paths (`{YYYY}/{MM}/{DD}/`) enabling efficient date-range queries and lifecycle management.

### Scenario 4: Archive write failure does not drop the message
**Given** a transient GCS write error occurs
**When** the archive write fails
**Then** the system retries the GCS write up to 3 times with exponential backoff; if all retries fail, the message is written to a local fallback queue and an alert fires; the ACK is still sent after successful fallback write.

---

## Definition of Done

- [ ] GCS write using `google-cloud-storage` SDK occurs atomically before ACK response is dispatched
- [ ] Archive path: `gs://{HIPAA_BUCKET}/{YYYY}/{MM}/{DD}/{msg_control_id}.hl7`
- [ ] Idempotency check: query `adt_event` table by `source_message_id` (MSH-10) before processing; return early if found
- [ ] `source_message_id` is indexed on `adt_event` table for fast lookup (O(log n))
- [ ] GCS write retries: 3 attempts, exponential backoff (1s, 2s, 4s)
- [ ] Unit tests: (a) archive path format, (b) duplicate detection, (c) GCS failure retry
- [ ] HIPAA bucket has: object versioning enabled, CMEK, Uniform Bucket-Level Access, no public access
- [ ] Code reviewed and approved

---

## Technical Notes

- Use `google.cloud.storage.Client.bucket(BUCKET_NAME).blob(path).upload_from_string(raw_hl7)` for atomic single-part upload
- MSH-10 from `hl7apy`: `message.MSH.MSH_10.value`
- Fallback queue: local in-memory deque (bounded) with background flush task; not a Redis dependency for Sprint 1
- Do NOT include patient PHI in the GCS object metadata — only message ID and timestamp in metadata
